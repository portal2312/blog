응용 프로그램 자체는 매우 간단하므로 코드가하는 일에 지나치게 주의를 기울이지 않아도 됩니다.

결국 Docker의 가치는 응용 프로그램을 작성, 배송 및 실행할 수있는 방법에 있습니다.

그것은 당신의 어플리케이션이 실제로하는 것에 관해서는 전혀 몰랐다.

Prerequisites

길을 따라 개념을 정의하지만, Docker가 무엇인지, Docker를 사용하기 전에 왜 사용해야 하는지를 이해하는 것이 좋습니다.

계속하기 전에 몇 가지 개념에 익숙하다고 가정해야합니다.

  • IP 주소와 Port
  • Virtual Machines (가성머신)
  • 구성 파일 편집
  • 코드 의존성 및 빌딩 아이디어에 대한 기본 지식
  • CPU 사용률, 바이트 단위의 RAM 사용 등과 같은 컴퓨터 리소스 사용 조건

마지막으로 Docker ID에 가입하고 로컬 컴퓨터에서 다음 명령을 실행하여 사용하면 이러한 일이 필요할 때 다시 생각 나게 할 수 있습니다.

$ docker login

A brief explanation of containers

Image 는 코드, 런타임, 라이브러리, 환경 변수 및 구성 파일을 포함하여 소프트웨어를 실행하는 데 필요한 모든 것을 포함하는 가벼운 독립 실행형 패키지입니다.

Container 는 images 의 런타임 instance 이며 실제로 실행될 때 images 가 메모리에 저장됩니다. 기본적으로 호스트 환경과 완벽하게 격리되어 실행되므로 호스트 파일과 포트에 액세스하는 경우에만 액세스합니다.

Container 는 기본적으로 호스트 시스템의 커널에서 응용 프로그램을 실행합니다.

하이퍼 바이저를 통해 호스트 리소스에 가상 액세스 만하는 가상 시스템보다 성능 특성이 우수합니다.

Container 는 고유한 액세스 권한을 가질 수 있습니다.

각 액세스 권한은 개별 프로세스에서 실행되며 다른 실행 파일보다 더 많은 메모리를 차지하지 않습니다.

Containers vs. virtual machines

Virtual Machine diagram

가상 컴퓨터는 게스트 운영 체제를 실행합니다. 각 상자의 OS 계층을 확인하십시오. 자원 집약적인 결과 디스크 이미지 및 응용 프로그램 상태는 OS 설정, 시스템 설치 종속성, OS 보안 패치 및 기타 쉽게 잃을 수 있고 복제하기가 쉬운 디스크를 모방한 것입니다.

Container diagram

Container 는 단일 커널을 공유할 수 있으며 컨테이너 이미지에 있어야하는 정보는 실행 파일과 패키지 종속성이며 Host 시스템에 절대 설치할 필요가 없습니다. 이러한 프로세스는 네이티브 프로세스와 같이 실행되며, Linux에서 ps를 실행하여 활성 프로세스를 확인하는 것처럼 docker ps와 같은 명령을 실행하여 개별적으로 프로세스를 관리 할 수 ​​있습니다. 마지막으로, 모든 종속성을 포함하기 때문에 구성 얽힘이 없습니다. 컨테이너화 된 앱은 "어디에서나 실행됩니다."

Conclustion

개인 단위의 실행 가능한 규모의 단위는 광범위한 영향을 미칩니다.

즉, CI / CD가 분산 응용 프로그램의 모든 부분에 업데이트를 푸시할 수 있으며 시스템 종속성은 문제가되지 않으며 자원 밀도가 증가한다는 의미입니다.

크기 조정 동작을 조정하면 새로운 VM 호스트가 아닌 새로운 실행 파일을 생성 할 수 있습니다.
