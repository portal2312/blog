Introduction

Part 3 에서는 Part 2에서 작성한 응용 프로그램을 가져 와서 service 로 전환하여 production 환경에서 실행해야하는 방법을 정의하고 프로세스에서 5 배로 확장했습니다. 여기 Part 4 에서는 이 응용 프로그램을 여러 컴퓨터에서 실행하는 클러스터에 배포합니다. 여러 컨테이너, 다중 기계 응용 프로그램은 여러 대의 컴퓨터를 swarm 라고 불리는 "Dockerized" 클러스터로 결합하여 가능합니다.

Understanding Swarm clusters

swarm 는 Docker를 실행하고 클러스터에 합류한 컴퓨터 그룹입니다. 그런 일이 발생하면, 이전에는 사용하던 Docker 명령을 계속 실행하지만, 이제는 swarm manager 가 클러스터에서 실행합니다. swarm 에 있는 machine 은 물리적 또는 가상일 수 있습니다. swarm 에 joining 시 node 라고 부릅니다.

Swarm managers 는 활용도가 가장 낮은 machine 을 container 로 채우는 "emptiest node"와 같은 container 를 실행하기 위한 몇 가지 전략을 사용할 수 있습니다. 또는 "Global". 각 기계가 지정된 container 의 instance 를 정확히 하나씩 가져옵니다. swarm managers 가 이미 사용하고 있는 것과 마찬가지로 작성 전략에서 이 전략을 사용하도록 지시합니다.

Swarm managers 는 명령을 실행할 수 있는 swarm 내의 유일한 기계이거나 다른 기계가 swarm에 Workers 로 참여할 수 있는 권한을 부여합니다. Workers 는 수용 능력을 제공하기 위해 존재하며, 다른 기계에 무엇을 할 수 있고 할 수 없는지를 말할 권한이 없습니다.

지금까지 로컬 시스템에서 단일 호스트 모드로 Docker를 사용해 왔다. 그러나 Docker는 swarm mode 로 전환 될 수 있으며, 이것이 Swarm의 사용을 가능하게 합니다. swarm mode 를 즉시 활성화하면 현재 시스템이 swarm managers 가 됩니다. 그때부터 Docker는 현재 컴퓨터에서만 관리하는 것이 아니라 관리중인 명령에서 실행하는 명령을 실행합니다.

swarm 는 여러 nodes 로 구성되며 물리적 또는 가상 시스템이 될 수 있습니다. 기본 개념은 간단합니다: swarm mode 를 활성화하고 현재 시스템을 swarm manager 로 만들려면 docker swarm mode실행한 다음, 다른 machines 에서 workers 로 swarm 에 합류하게 docker swarm join 하십시오. 아래에서 탭을 선택하여 다양한 상황에서 이것이 어떻게 진행되는지 확인하십시오. 우리는 VM을 사용하여 2대의 machines cluster 를 신속하게 만들고 그것을 swarm 로 바꿀 것입니다.

INITIALZE THE SWARM AND ADD NODES

첫 번째 장비는 관리자로 작동하여 관리 명령을 실행하고 worker를 인증하여 swarm 에 가입시키며, 두 번째 시스템은 worker 가 됩니다.

docker-machine ssh 를 사용하여 VM에 명령을 보낼 수 있습니다. myvm1 에게 docker swarm init 가 있는 swarm manager가 되도록 지시하면 다음과 같은 출력이 표시됩니다:

$ docker-machine ssh myvm1 "docker swarm init --advertise-addr <myvm1 ip>"
# Swarm initialized: current node <node ID> is now a manager.
#
# To add a worker to this swarm, run the following command:
#
#   docker swarm join \
#   --token <token> \
#   <myvm ip>:<port>
#
# To add a manager to this swarm, run 'docker swarm join-token manager' and follow the instructions.

예:

$ docker-machine ssh myvm1 "docker swarm init --advertise-addr 192.168.99.100:2377"

Ports 2377 and 2376

docker swarm init and docker swarm join 는 port 2377 (swarm mangement port)를 항상 실행하거나, 전혀 포트를 사용하지 않고 기본값을 사용합니다.

docker-machine ls 에 의해 반환된 컴퓨터 IP 주소는 Docker 데몬 port인 포트 2376 을 포함합니다.

이 포트를 사용하지 마십시오. 오류가 발생할 수 있습니다.

보시다시피 docker swarm init 에 대한 응답에는 추가하려는 node 에서 실행할 수 있도록 미리 구성된 docker swarm join 명령어가 포함되어 있습니다. 이 명령어을 복사하여 docker-machine ssh 를 통해 myvm2 로 보내면 myvm2 가 새 swarm 을 worker 로 참여시킵니다.

$ docker-machine ssh myvm2 "docker swarm join \
--token <token> \
<ip>:2377"

Configure a docker-machine shell to the swarm manager

지금까지 docker-machine ssh 에서 Docker 명령을 랩핑하여 VM과 대화했습니다. 또 다른 옵션은 docker-machine env <machine> 을 실행하여 현재 shell 을 구성하여 VM의 Docker 데몬과 통신하는 명령을 실행하는 것입니다. 이 방법은 로컬 docker-compose.yml 파일을 사용하여 어디서나 복사하지 않고 "원격"으로 app 을 배포 할 수 있기 때문에 보다 다음 단계에서 효과적입니다.

docker-machine env myvm1 를 입력하고, 출력의 마지막 줄에 제공된 명령을 복사하여 붙여넣고 실행하여 shell 이 swarm manager 인 myvm1 과 통신하도록 구성하십시오.

Shell 을 구성하는 명령은 Mac, Linux 또는 Windows 중 어떤 것이 있는지에 따라 다르므로 아래 각 탭에 예제가 나와 있습니다.

DOCKER MACHINE SHELL ENVIRONMENT ON MAC OR LINUX

docker-machine env myvm1 을 실행하여 myvm1 과 통신하도록 shell 을 구성하는 명령을 얻으십시오.

$ docker-machine env myvm1

export DOCKER_TLS_VERIFY="1"
export DOCKER_HOST="tcp://192.168.99.100:2376"
export DOCKER_CERT_PATH="/home/docker/.docker/machine/machines/myvm1"
export DOCKER_MACHINE_NAME="myvm1"
# Run this command to configure your shell:
# eval $(docker-machine env myvm1)

주어진 명령을 실행하여 shell 이 myvm1 과 통신하도록 구성하십시오.

$ eval $(docker-machine env myvm1)

docker-machine ls 를 실행하여 옆에 asterisk(*) 가 표시된대로 myvm1 이 현재 활성 시스템인지 확인하십시오.

$ docker-machine ls

# NAME    ACTIVE   DRIVER       STATE     URL                         SWARM   DOCKER        ERRORS
# myvm1   *        virtualbox   Running   tcp://192.168.99.100:2376           v17.11.0-ce
# myvm2   -        virtualbox   Running   tcp://192.168.99.101:2376           v17.11.0-ce

Connecting to VMs with docker-machine env and docker-machine ssh

  • myvm2 처럼 다른 machine 과 대화할 shell 을 설정하려면, 동일하거나 다른 shell 에서 docker-machine env 를 다시 실행 한 다음 주어진 명령을 실행하여 myvm2 를 가리키십시오. 이것은 항상 현재 shell 에만 적용됩니다. 구성되지 않은 shell 로 변경하거나 new shell 을 열면 명령을 다시 실행해야합니다. docker-machine ls 를 사용하여 기계를 나열하고, 그들이 어떤 상태인지 파악하고, IP ​​주소를 얻고, 연결되어있는 경우 어떤 것이 있는지 찾아보십시오. 자세한 내용은 Docker Machine getting started topics 참조하십시오.

  • 또는 Docker 명령을 docker-machine ssh <machine> "<command>" 형식으로 래핑할 수도 있습니다. 이 명령은 VM에 직접 기록하지만 로컬 호스트의 파일에 즉시 액세스 할 수는 없습니다.

  • Mac 및 Linux에서는 docker-machine scp <file> <machine>:~ 을 사용하여 여러 컴퓨터에서 파일을 복사 할 수 있지만 Windows 사용자는이 작업을 위해서는 Git Bash와 같은 Linux 터미널 에뮬레이터가 필요합니다.

이 튜토리얼은 docker-machine CLI 를 통해 모든 플랫폼에서 사용할 수 있으므로 docker-machine ssh 와 docker-machine env 를 모두 데모합니다.

Accessing your cluster

myvm1 또는 myvm2의 IP 주소에서 앱에 액세스 할 수 있습니다.

사용자가 만든 네트워크는 네트워크간에 공유되며 load-balancing 을 수행합니다. docker-machine ls 를 실행하여 VM의 IP 주소를 얻고 브라우저에서 해당 주소로 방문하여 새로 고침 (또는 curl)하십시오.

5 개의 가능한 컨테이너 ID가 모두 무작위로 순환하여 load-balancing을 보여줍니다.

두 IP 주소가 작동하는 이유는 한 무리의 node 가 수신 routing mesh 에 참여한다는 것입니다. 이렇게하면 실제로 container 를 실행중인 node 가 무엇이든 관계없이 swarm의 특정 포트에 배포 된 서비스가 항상 해당 port 를 소유하도록 할 수 있습니다. 다음 다이어그램은 3 node 뭉치에서 포트 8080 에 게시된 my-web 이라는 서비스용 routing mesh 가 어떻게 보이는지 입니다.

Having connectivity trouble?

swarm mode 활성화하기 전에 swarm 에 진입 네트워크를 사용하려면 swarm 노드간 다음 Port 를 열어야 합니다. - container 네트워크 검색을 위한 Port 7946 TCP/UDP - container 수신용 네트워크를 위한 Port 4789 UDP

Iterating and scaling your app

여기에서 파트 2와 3에서 배운 모든 것을 할 수 있습니다.

docker-compose.yml 파일을 변경하여 응용 프로그램의 크기를 조정하십시오.

코드를 편집하여 응용 프로그램 동작을 변경 한 다음 새 이미지를 다시 작성하고 푸시합니다. (이렇게하려면 이전에 앱을 제작하고 이미지를 게시하는 데 사용한 것과 동일한 단계를 따르십시오).

Understanding Swarm clusters

swarm 는 Docker를 실행하고 클러스터에 합류한 컴퓨터 그룹입니다. 그런 일이 발생하면, 이전에는 사용하던 Docker 명령을 계속 실행하지만, 이제는 swarm manager 가 클러스터에서 실행합니다. swarm 에 있는 machine 은 물리적 또는 가상일 수 있습니다. swarm 에 joining 시 node 라고 부릅니다.

Swarm managers 는 활용도가 가장 낮은 machine 을 container 로 채우는 "emptiest node"와 같은 container 를 실행하기 위한 몇 가지 전략을 사용할 수 있습니다. 또는 "Global". 각 기계가 지정된 container 의 instance 를 정확히 하나씩 가져옵니다. swarm managers 가 이미 사용하고 있는 것과 마찬가지로 작성 전략에서 이 전략을 사용하도록 지시합니다.

Swarm managers 는 명령을 실행할 수 있는 swarm 내의 유일한 기계이거나 다른 기계가 swarm에 Workers 로 참여할 수 있는 권한을 부여합니다. Workers 는 수용 능력을 제공하기 위해 존재하며, 다른 기계에 무엇을 할 수 있고 할 수 없는지를 말할 권한이 없습니다.

지금까지 로컬 시스템에서 단일 호스트 모드로 Docker를 사용해 왔다. 그러나 Docker는 swarm mode 로 전환 될 수 있으며, 이것이 Swarm의 사용을 가능하게 합니다. swarm mode 를 즉시 활성화하면 현재 시스템이 swarm managers 가 됩니다. 그때부터 Docker는 현재 컴퓨터에서만 관리하는 것이 아니라 관리중인 명령에서 실행하는 명령을 실행합니다.

  • myvm2 처럼 다른 machine 과 대화할 shell 을 설정하려면, 동일하거나 다른 shell 에서 docker-machine env 를 다시 실행 한 다음 주어진 명령을 실행하여 myvm2 를 가리키십시오. 이것은 항상 현재 shell 에만 적용됩니다. 구성되지 않은 shell 로 변경하거나 new shell 을 열면 명령을 다시 실행해야합니다. docker-machine ls 를 사용하여 기계를 나열하고, 그들이 어떤 상태인지 파악하고, IP ​​주소를 얻고, 연결되어있는 경우 어떤 것이 있는지 찾아보십시오. 자세한 내용은 Docker Machine getting started topics 참조하십시오.

  • 또는 Docker 명령을 docker-machine ssh <machine> "<command>" 형식으로 래핑할 수도 있습니다. 이 명령은 VM에 직접 기록하지만 로컬 호스트의 파일에 즉시 액세스 할 수는 없습니다.

  • Mac 및 Linux에서는 docker-machine scp <file> <machine>:~ 을 사용하여 여러 컴퓨터에서 파일을 복사 할 수 있지만 Windows 사용자는이 작업을 위해서는 Git Bash와 같은 Linux 터미널 에뮬레이터가 필요합니다.

이 튜토리얼은 docker-machine CLI 를 통해 모든 플랫폼에서 사용할 수 있으므로 docker-machine ssh 와 docker-machine env 를 모두 데모합니다.

Accessing your cluster

myvm1 또는 myvm2의 IP 주소에서 앱에 액세스 할 수 있습니다.

사용자가 만든 네트워크는 네트워크간에 공유되며 load-balancing 을 수행합니다. docker-machine ls 를 실행하여 VM의 IP 주소를 얻고 브라우저에서 해당 주소로 방문하여 새로 고침 (또는 curl)하십시오.

5 개의 가능한 컨테이너 ID가 모두 무작위로 순환하여 load-balancing을 보여줍니다.

두 IP 주소가 작동하는 이유는 한 무리의 node 가 수신 routing mesh 에 참여한다는 것입니다. 이렇게하면 실제로 container 를 실행중인 node 가 무엇이든 관계없이 swarm의 특정 포트에 배포 된 서비스가 항상 해당 port 를 소유하도록 할 수 있습니다. 다음 다이어그램은 3 node 뭉치에서 포트 8080 에 게시된 my-web 이라는 서비스용 routing mesh 가 어떻게 보이는지 입니다.

Having connectivity trouble?

swarm mode 활성화하기 전에 swarm 에 진입 네트워크를 사용하려면 swarm 노드간 다음 Port 를 열어야 합니다. - container 네트워크 검색을 위한 Port 7946 TCP/UDP - container 수신용 네트워크를 위한 Port 4789 UDP

Cleanup and reboot

Stacks and swarms

docker stack rm 을 사용하여 stack 을 삭제할 수 있습니다. 예:

$ docker stack rm getstartedlab

Keep the swarm or remove it?

나중에 관리자에서 docker-machine ssh myvm2 "docker swarm leave" 와 docker-machine ssh myvm1 "docker swarm leave --force" 이용해서 원한다면 swarm 를 제거 할 수 있습니다. 그러나 이 swarm 은 part 5 필요하니 보관하십시오.

Unsetting docker-machine shell variable settings

다음 명령을 사용하여 현재 쉘에서 docker-machine 환경 변수를 설정 해제 할 수 있습니다.

eval $(docker-machine env -u)

이렇게 하면 docker 컴퓨터에서 만든 가상 컴퓨터와 셸의 연결이 끊어지며 동일한 셸에서 작업을 계속할 수 있습니다. 예를 들어 Mac 용 Docker 또는 Windows 용 Docker와 같은 기본 docker 명령을 사용합니다. 자세한 내용은 Machine topic on unsetting environment variables 참조.

Restarting Docker machines

만일 local host 를 종료할 때, Docker machines 중지됩니다. docker-machine ls 를 실행하여 machines 의 상태를 확인할 수 있습니다.

$ docker-machine ls

재시작하기:

$ docker-machine start <machine-name>

part 4 에서는 swarm 이 무엇인지, swarm node 가 관리자 또는 작업자가 될 수 있는지, swarm 을 만들어서 어떻게 응용 프로그램을 배포 하는지를 배웠습니다. Docker 핵심 명령이 part 3에서 변경되지 않은 것을 보았을 때, 그들은 swarm master 는 목표 향해 달려 가야했습니다. Docker의 네트워킹 기능이 실제로 다른 시스템에서 실행 중이더라도 container 간에 load-balancing request 을 보냈습니다. 마지막으로 cluster 에서 app을 반복하고 크기를 조정하는 방법을 배웠습니다.

다음은 swarm 및 VM과 상호 작용하기 위해 실행할 수있는 몇 가지 명령입니다.

docker-machine create --driver virtualbox myvm1 # Create a VM (Mac, Win7, Linux)
docker-machine create -d hyperv --hyperv-virtual-switch "myswitch" myvm1 # Win10
docker-machine env myvm1                # View basic information about your node
docker-machine ssh myvm1 "docker node ls"         # List the nodes in your swarm
docker-machine ssh myvm1 "docker node inspect <node ID>"        # Inspect a node
docker-machine ssh myvm1 "docker swarm join-token -q worker"   # View join token
docker-machine ssh myvm1   # Open an SSH session with the VM; type "exit" to end
docker node ls                # View nodes in swarm (while logged on to manager)
docker-machine ssh myvm2 "docker swarm leave"  # Make the worker leave the swarm
docker-machine ssh myvm1 "docker swarm leave -f" # Make master leave, kill swarm
docker-machine ls # list VMs, asterisk shows which VM this shell is talking to
docker-machine start myvm1            # Start a VM that is currently not running
docker-machine env myvm1      # show environment variables and command for myvm1
eval $(docker-machine env myvm1)         # Mac command to connect shell to myvm1
& "C:\Program Files\Docker\Docker\Resources\bin\docker-machine.exe" env myvm1 | Invoke-Expression   # Windows command to connect shell to myvm1
docker stack deploy -c <file> <app>  # Deploy an app; command shell must be set to talk to manager (myvm1), uses local Compose file
docker-machine scp docker-compose.yml myvm1:~ # Copy file to node's home dir (only required if you use ssh to connect to manager and deploy the app)
docker-machine ssh myvm1 "docker stack deploy -c <file> <app>"   # Deploy an app using ssh (you must have first copied the Compose file to myvm1)
eval $(docker-machine env -u)     # Disconnect shell from VMs, use native docker
docker-machine stop $(docker-machine ls -q)               # Stop all running VMs
docker-machine rm $(docker-machine ls -q) # Delete all VMs and their disk images
