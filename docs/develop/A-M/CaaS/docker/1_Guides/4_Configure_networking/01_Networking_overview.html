Docker 컨테이너와 서비스가 강력한 이유 중 하나는 Docker 컨테이너와 서비스를 서로 연결하거나 비 Docker 작업 부하에 연결할 수 있다는 것입니다. Docker 컨테이너 및 서비스는 Docker에 배포되어 있는지 또는 동료가 Docker 작업 부하인지 여부도 알 필요가 없습니다. Docker 호스트가 Linux, Windows 또는이 둘을 혼합하여 실행하는 경우에도 Docker를 사용하여 플랫폼에 구애받지 않는 방식으로 Docker를 관리 할 수 ​​있습니다.

Scope of this topic

이 항목에서는 Docker 네트워크 작동 방식에 대한 OS 별 세부 정보를 다루지 않으므로 Linux에서 Docker가 iptables 규칙을 조작하는 방법이나 Windows 서버에서 라우팅 규칙을 처리하는 방법에 대한 정보는 찾을 수 없으며 Docker가 어떻게 작동하는지에 대한 자세한 정보는 찾을 수 없습니다 패킷을 형성하고 캡슐화하거나 암호화를 처리합니다.

Docker 및 iptables 및 Docker Reference Architecture : 기술 세부 사항에 대한 자세한 내용은 확장 가능한 휴대용도 커 컨테이너 네트워크 설계를 참조하십시오.

또한 이 항목에서는 Docker 네트워크를 만들고, 관리하고, 사용하는 방법에 대한 자습서를 제공하지 않습니다.

각 섹션에는 관련 자습서 및 명령 참조에 대한 링크가 포함되어 있습니다.

Network drivers

Docker의 네트워킹 하위 시스템은 드라이버를 사용하여 플러그 가능합니다. 기본적으로 여러 드라이버가 존재하며 핵심 네트워킹 기능을 제공합니다.

  • bridge:

    • 기본 네트워크 드라이버.
    • 드라이버를 지정하지 않으면 작성중인 네트워크 유형입니다.
    • 브리지 네트워크는 대개 응용 프로그램이 통신해야하는 독립 실행 형 컨테이너에서 실행될 때 사용됩니다. bridge networks 참조.

  • host:

  • overlay:

  • macvlan:

  • none:

  • Network plugins:

Network driver summary

  • 사용자 정의 브리지 네트워크는 여러 Docker가 동일한 Docker 호스트에서 통신해야하는 경우에 가장 적합합니다.

  • 호스트 네트워크는 Docker 호스트에서 네트워크 스택을 격리해서는 안되지만 컨테이너의 다른 측면을 격리하려고 할 때 가장 좋습니다.

  • 오버레이 네트워크는 서로 다른 Docker 호스트에서 실행되는 컨테이너가 통신 할 필요가 있거나 웜 서비스를 사용하여 여러 응용 프로그램이 함께 작동 할 때 가장 좋습니다.

  • Macvlan 네트워크는 VM 설정에서 마이그레이션하거나 컨테이너가 고유 한 MAC 주소를 가진 네트워크의 실제 호스트처럼 보이도록 할 때 가장 좋습니다.

  • 타사 네트워크 플러그인을 사용하면 Docker를 특수 네트워크 스택과 통합 할 수 있습니다.
