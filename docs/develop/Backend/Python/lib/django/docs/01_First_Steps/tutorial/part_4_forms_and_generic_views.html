Write a simple form

<!--polls/templates/polls/detail.html-->

<h1>{{ question.question_text }}</h1>

{% if error_message %}<p><strong>{{ error_message }}</strong></p>{% endif %}

<form action="{% url 'polls:vote' question.id %}" method="post">
{% csrf_token %}
{% for choice in question.choice_set.all %}
    <input
        type="radio"
        name="choice"
        id="choice{{ forloop.counter }}"
        value="{{ choice.id }}"
    />
    <label for="choice{{ forloop.counter }}">
        {{ choice.choice_text }}
    </label>
    <br />
{% endfor %}
<input type="submit" value="Vote" />
</form>

  • choice=# : 선택 choice ID, HTML 기본코드

  • method="post" : server side 로 data 전송에서 사용. (반대 method="get")

  • forloop.counter : for문을 몇번 통과했는지. choice{{ forloop.counter }}

  • {\% csrf_token \%}

POST 양식 (Data 수정의 영향을 미칠 수 있음)

Cross Site Request Forgeies(요청 위조)에 대해 걱정할 필요가 있음.

내부 URL을 대상으로하는 모든 POST양식은 {\% csrf_token \%} 템플릿 태그를 사용해야 함.

# polls/urls.py

url(r'^(?P<question_id>[0-9]+)/vote/$', views.vote, name='vote'),

# polls/views.py

from django.shortcuts import get_object_or_404, render
from django.http import HttpResponseRedirect, HttpResponse
from django.urls import reverse

from .models import Choice, Question
# ...
def vote(request, question_id):
    question = get_object_or_404(Question, pk=question_id)
    try:
        selected_choice = question.choice_set.get(pk=request.POST['choice'])
    except (KeyError, Choice.DoesNotExist):
        # Redisplay the question voting form.
        return render(request, 'polls/detail.html', {
            'question': question,
            'error_message': "You didn't select a choice.",
        })
    else:
        selected_choice.votes += 1
        selected_choice.save()
        # Always return an HttpResponseRedirect after successfully dealing
        # with POST data. This prevents data from being posted twice if a
        # user hits the Back button.
        return HttpResponseRedirect(reverse('polls:results', args=(question.id,)))

상세설명:

  • request.POST

key 이름으로 제출 된 데이터에 액세스할 수 있는 사전형 객체 임.

이 경우 request.POST['choice'] 는 선택한 선택 항목의 ID를 문자열로 반환.

request.POST 값은 항상 문자열 임.

(Django는 또한 동일한 방식으로 데이터에 접근하기 위한 request.GET 을 제공)

(POST 호출을 통해서만 데이터가 변경되도록 코드에서 명시적으로 사용)

(추가설명: Django가 REST API를 지향, GET - 가져오기 / POST - 보내기)

  • request.POST['choice']

POST 데이터에 선택 사항이 제공되지 않은 경우 KeyError를 발생 시킴.

KeyError를 검사하고 선택사항이 제공되지 않으면 오류 메시지와 함께 질문 양식을 다시 표시.

  • HttpResponseRedirect

POST 데이터를 성공적으로 처리 한 후에는 항상 HttpResponseRedirect를 리턴하십시오.

이렇게 하면 사용자가 뒤로 버튼을 누르면 데이터가 두 번 게시되지 않습니다.

선택 횟수를 증가 시키면 코드는 일반 HttpResponse가 아닌 HttpResponseRedirect를 반환.

사용자가 리디렉션 되는 URL(이 경우 URL을 구성하는 방법에 대한 다음 부분 참조)이라는 단일 인수를 취함.

  • reverse()

view 함수에서 URL을 하드코딩하지 않아도 됨.

여기에는 제어를 전달하려는 view의 이름과 해당보기를 가리키는 URL패턴의 가변부분이 제공 됨.

이 경우, 설정된 URLconf를 사용하여 이 reverse() 호출은 다음과 같은 문자열을 리턴 함.

  '/polls/3/results/'

/3/ == question.id 임. 이렇게 리디렉션된 URL은 vote() 보기를 호출하여 최종 페이지를 표시.

request는 HTTPRequest의 객체.

누군가가 질문에 투표하면 vote() 보기가 질문의 결과 페이지로 리디렉션 됨.

# polls/views.py
from django.shortcuts import get_object_or_404, render


def results(request, question_id):
    question = get_object_or_404(Question, pk=question_id)
    return render(request, 'polls/results.html', {'question': question})

참조: Avoiding race conditions using F()

Use generic views: Less code is better

views를 기본적인 웹 개발의 공통 case로 재선언:

URL에서 전달 된 매개 변수에 따라 Database에서 get Data, template load 및 render된 template return.

Django는 매우 일반적이기 때문에 "일반 view" 시스템이라는 바로 가기를 제공함.

일반 뷰는 앱을 작성하기 위해 파이썬 코드를 작성하지 않아도되는 일반적인 패턴을 추상화 함.

apps을 일반 views 시스템을 사용하도록 변환하여 우리 자신의 코드를 삭제할 수 있도록 함:

  1. URLconf 로 변환
  2. 삭제 - 오래된 일부, 불필요한 views
  3. 도입 - 장고의 일반적인 views에 기초한 새로운 views

자세한 내용 읽기

* Why the code-shuffle?

일반적으로 Django 응용 프로그램을 작성할 때,
일반 views가 문제에 적합한 지 여부를 평가하고
코드를 중간에서 리팩토링하지 않고 처음부터 사용됨.

그러나 이 튜토리얼은 핵심 개념에 초점을 맞추기 위해 고의적으로
현재까지 "어려운 방법"으로 의견을 작성하는 데 중점을 두고 있음.

계산기를 사용하기 전에 기본 수학을 알아야 함.

Amend URLconf

URLconf 개선

# polls/urls.py

from django.conf.urls import url

from . import views

app_name = 'polls'
urlpatterns = [
    url(r'^$', views.IndexView.as_view(), name='index'),
    url(r'^(?P<pk>[0-9]+)/$', views.DetailView.as_view(), name='detail'),
    url(r'^(?P<pk>[0-9]+)/results/$', views.ResultsView.as_view(), name='results'),
    url(r'^(?P<question_id>[0-9]+)/vote/$', views.vote, name='vote'),
]

regex 패턴 변경 : --> <pk>

view 변경 : views.index --> views.IndexView.as_view()

Amend views

views 개선

index, detail, results 이전의 views를 제거하고 Django의 generic views 사용.

# polls/views.py
from django.shortcuts import get_object_or_404, render
from django.http import HttpResponseRedirect
from django.urls import reverse
from django.views import generic

from .models import Choice, Question


class IndexView(generic.ListView):
    template_name = 'polls/index.html'
    context_object_name = 'latest_question_list'

    def get_queryset(self):
        """Return the last five published questions."""
        return Question.objects.order_by('-pub_date')


class DetailView(generic.DetailView):
    model = Question
    template_name = 'polls/detail.html'


class ResultsView(generic.DetailView):
    model = Question
    template_name = 'polls/results.html'


def vote(request, question_id):
    ... # same as above, no changes needed.

generic views (현재 사용된 두 가지만)

  • ListView : "개체 목록 표시"

  • DetailView : "특정 개체 유형에 대한 세부 정보 페이지 표시"

generic views 사용전

  • 어떤 모델이 적용될 것인지를 알아야 함. model 속성을 사용하여 제공 되기 때문.

  • DetailView generic view는 URL에서 캡처 한 primary key 값이 "pk"라고 예상하므로 generic 뷰의 경우 question_id를 pk로 변경.

특징

  • template 명 지정

    • default template 명
      • ListView : <app name>/<model name>_list.html
      • DetailView : <app name>/<model name>_detail.html
    • 수동으로 변경가능
      template_name = 'polls/index.html'

  • variables 자동 제공(생성)

    Django model을 사용하고 있기 때문에 스스로 적절한 context 변수명을 결정.

    • Example

      • DetailView : question
      • ListView : question_list

      자동 생성된 question_list context 변수

      대신 context_object_name 속성을 제공하여 latest_question_list 를 대신 사용.

      context_object_name = 'latest_question_list'

    • 적용 방법

      • (추천) Django에 변경하는 것이 훨씬 쉬움.
      • templates을 새로운 기본 context 변수와 일치하도록 변경할 수 있음.

서버를 실행하고 일반보기를 기반으로 새 폴링 앱을 사용하십시오.

generic views에 대한 자세한 내용은 generic views documentation를 참조.
