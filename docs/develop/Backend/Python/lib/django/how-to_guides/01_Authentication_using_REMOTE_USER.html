Authentication using REMOTE_USER

Table of contents

  • Configuration
  • Using REMOTE_USER on login pages only
  • References

이 문서는 Django 응용 프로그램에서 외부 인증 소스 (웹 서버가 REMOTE_USER 환경 변수를 설정하는 곳)를 사용하는 방법을 설명합니다.

이러한 유형의 인증 솔루션은 일반적으로 IIS 및 Windows 통합 인증 또는 Apache, mod_authnz_ldap, CAS, Cosign, WebAuth, mod_auth_sspi 등과 같은 단일 사인온 솔루션을 사용하여 인트라넷 사이트에 표시됩니다.

웹 서버가 인증을 처리 할 때 일반적으로 기본 응용 프로그램에서 사용할 REMOTE_USER 환경 변수를 설정합니다. Django에서 REMOTE_USER는 request.META 속성에서 사용할 수 있습니다. Django는 RemoteUserMiddleware 또는 PersistentRemoteUserMiddleware 및 django.contrib.auth에 있는 RemoteUserBackend 클래스를 사용하여 REMOTE_USER 값을 사용하도록 구성 할 수 있습니다.

Configuration

먼저, MIDDLEWARE 설정에서 django.contrib.auth.middleware.AuthenticationMiddleware 뒤에 django.contrib.auth.middleware.RemoteUserMiddleware 를 추가해야 합니다:

MIDDLEWARE = [
    '...',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.auth.middleware.RemoteUserMiddleware',
    '...',
]

그 다음, AUTHENTICATION_BACKENDS 설정에서 ModelBackend를 RemoteUserBackend로 대체해야 합니다:

AUTHENTICATION_BACKENDS = [
    'django.contrib.auth.backends.RemoteUserBackend',
]

이 설정을 사용하면 RemoteUserMiddleware가 request.META['REMOTE_USER']에서 사용자 이름을 감지하고 RemoteUserBackend를 사용하여 해당 사용자를 인증하고 자동 로그인합니다.

이 특정 설정은 기본 ModelBackend로 인증을 비활성화합니다. 이것은 REMOTE_USER 값이 설정되어 있지 않으면 Django의 admin 인터페이스를 사용해도 사용자가 로그인 할 수 없다는 것을 의미합니다. AUTHENTICATION_BACKENDS 목록에 'django.contrib.auth.backends.ModelBackend'를 추가하면 REMOTE_USER가 없는 경우 ModelBackend가 대체되어 이러한 문제가 해결됩니다.

contrib.admin 및 createsuperuser 관리 명령의 보기와 같은 django 사용자 관리는 원격 사용자와 통합되지 않습니다. 이러한 인터페이스는 AUTHENTICATION_BACKENDS와 관계없이 데이터베이스에 저장된 사용자와 함께 작동합니다.

Note

RemoteUserBackend는 ModelBackend에서 상속되므로 ModelBackend에 구현 된 것과 동일한 권한 검사를 여전히 갖게됩니다. is_active = False 인 사용자는 인증 할 수 없습니다. AllowAllUsersRemoteUserBackend를 허용하려면 허용하십시오.

인증 메커니즘이 REMOTE_USER가 아닌 사용자 정의 HTTP 헤더를 사용하는 경우 RemoteUserMiddleware를 서브 클래스화하고 헤더 속성을 원하는 request.META key로 설정할 수 있습니다. 예:

from django.contrib.auth.middleware import RemoteUserMiddleware

class CustomHeaderMiddleware(RemoteUserMiddleware):
    header = 'HTTP_AUTHUSER'

Warning

사용자 정의 HTTP 헤더와 함께 RemoteUserMiddleware 서브 클래스를 사용하는 경우 매우 주의하십시오. 프런트 엔드 웹 서버는 최종 사용자가 위조 된 (또는 "스푸핑 된") 헤더 값을 제출하지 못하도록 적절한 인증 검사를 기반으로 해당 헤더를 항상 설정하거나 제거해야합니다. HTTP 헤더 X-Auth-User와 X-Auth_User (예)는 모두 request.META의 HTTP_X_AUTH_USER key로 정상화되므로 대시 대신 밑줄을 사용하여 스푸핑 된 헤더를 허용하지 않는지 웹 서버에서 확인해야 합니다.

request.META에서 HTTP_로 시작하지 않는 키는 HTTP 요청 헤더에서 직접 가져 오는 것이 아니라 WSGI 서버에서만 설정할 수 있기 때문에 이 경고는 header='REMOTE_USER'인 기본 구성의 RemoteUserMiddleware 에는 적용되지 않습니다.

더 많은 제어가 필요한 경우 RemoteUserBackend 를 상속 한 자체 인증 백엔드를 작성하고 하나 이상의 특성 및 메소드를 대체 할 수 있습니다.

Using REMOTE_USER on login pages only

로그인 페이지에서만 REMOTE_USER 사용하기

RemoteUserMiddleware 인증 미들웨어는 HTTP 요청 헤더 REMOTE_USER가 모든 인증된 요청과 함께 존재한다고 가정합니다. htpasswd 또는 다른 간단한 메커니즘을 사용하는 기본 HTTP Auth가 사용되지만 Negotiate (GSSAPI/Kerberos) 또는 기타 자원 집약적인 인증 방법을 사용하면 프런트 엔드 HTTP 서버의 인증은 일반적으로 하나만 설정됩니다 또는 몇 개의 로그인 URL을 사용하고 인증 성공 후 응용 프로그램은 인증된 세션 자체를 유지 관리해야 합니다.

PersistentRemoteUserMiddleware는 이 사용 사례를 지원합니다. 사용자가 명시적으로 로그 아웃 할 때까지 인증 된 세션을 유지 관리합니다. 이 클래스는 위의 문서에서 RemoteUserMiddleware의 드롭인 대체품으로 사용할 수 있습니다.
