Introduction

Docker 방식으로 앱을 제작할 때입니다. 우리는 이 응용 프로그램의 계층 구조의 맨 아래부터 시작할 것입니다. 이 응용 프로그램은 컨테이너이며, 이 페이지에서 다루고 있습니다. 이 레벨 위에는 Part 3에서 다루는 컨테이너가 프로덕션 환경에서 작동하는 방식을 정의하는 서비스가 있습니다. 마지막으로 최상위 레벨에는 스택 (Part 5)의 모든 서비스의 상호 작용을 정의하는 스택이 있습니다.

You new development environment

과거에는 Python 응용 프로그램을 작성하기 시작한 경우 비즈니스의 첫 번째 순서는 Python 런타임을 시스템에 설치하는 것이 었습니다. 하지만, 이는 앱이 예상대로 실행되기 위해서는 컴퓨터 환경이 있어야 하는 상황을 만듭니다. 귀하의 application 을 실행하는 서버에 대해.

Docker를 사용하면 이식 가능한 Python 런타임을 이미지로 가져올 수 있으며 설치가 필요 없습니다. 그런 다음 빌드는 앱 코드와 함께 기본 Python 이미지를 포함 할 수 있으므로 앱, 종속성 및 런타임이 함께 ​​여행 할 수 있습니다.

이 휴대용 이미지는 Dockerfile 이라고하는 무언가에 의해 정의됩니다.

Define a container with a Dockerfile

Dockerfile 은 Container 내부의 환경을 정의합니다.

네트워킹 인터페이스 및 디스크 드라이브와 같은 리소스에 대한 액세스는 이 환경에서 가상화됩니다.

나머지 시스템과 격리되어 있으므로 포트를 외부 세계에 매핑해야 합니다.

그리고 해당 환경에 "복사"할 파일에 대해 구체적으로 설명하십시오.

그러나 이 작업을 수행하면 이 Dockerfile 에 정의된 앱 빌드가 어디서나 똑같이 작동합니다.

Dockerfile

빈 디렉토리를 만듭니다. 디렉토리 (cd)를 새 디렉토리로 변경하고 Dockerfile 이라는 파일을 작성한 후 다음 내용을 해당 파일에 복사하여 붙여넣고 저장하십시오. 새로운 Dockerfile의 각 문장을 설명하는 주석을 기록하십시오.

# Use an official Python runtime as a parent image
FROM python:2.7-slim

# Set the working directory to /app
WORKDIR /app

# Copy the current directory contents into the container at /app
ADD . /app

# Install any needed packages specified in requirements.txt
RUN pip install -r requirements.txt

# Make port 80 available to the world outside this container
EXPOSE 80

# Define environment variable
ENV NAME World

# Run app.py when the container launches
CMD ["python", "app.py"]

프록시 서버를 사용하고 있습니까?

프록시 서버는 일단 실행되면 웹 응용 프로그램에 대한 연결을 차단할 수 있습니다. 프록시 서버를 사용하는 경우 ENV 명령을 사용하여 프록시 서버의 호스트와 포트를 지정하여 다음 행을 Dockerfile에 추가하십시오.

# Set proxy server, replace host:port with values for your servers
ENV http_proxy host:port
ENV https_proxy host:port

이 Dockerfile은 아직 작성하지 않은 몇 가지 파일, 즉 app.py 및 requirements.txt를 참조합니다. 다음 단계를 만들어 보겠습니다.

The app itself

requirements.txt 와 app.py 파일을 두 개 더 만들고 Dockerfile 과 같은 폴더에 두십시오. 이것은 우리의 앱을 완성 시켰는데, 당신이 알듯이 아주 간단합니다. 위의 Dockerfile 이 이미지에 내장되면 Dockerfile의 ADD 명령으로 인해 app.py 및 requirements.txt 가 표시되고 EXPOSE 명령으로 인해 HTTP 를 통해 app.py 의 출력에 액세스 할 수 있습니다.

requirements.txt

Flask
Redis

app.py

from flask import Flask
from redis import Redis, RedisError
import os
import socket

# Connect to Redis
redis = Redis(host="redis", db=0, socket_connect_timeout=2, socket_timeout=2)

app = Flask(__name__)

@app.route("/")
def hello():
    try:
        visits = redis.incr("counter")
    except RedisError:
        visits = "<i>cannot connect to Redis, counter disabled</i>"

    html = "<h3>Hello {name}!</h3>" \
           "<b>Hostname:</b> {hostname}<br/>" \
           "<b>Visits:</b> {visits}"
    return html.format(name=os.getenv("NAME", "world"), hostname=socket.gethostname(), visits=visits)

if __name__ == "__main__":
    app.run(host='0.0.0.0', port=80)

이제 pip install -r requirements.txt 가 Python 용 Flask 및 Redis 라이브러리를 설치하고 app 이 socket.gethostname() 에 대한 호출 결과와 함께 환경 변수 NAME을 인쇄하는지 확인합니다.

마지막으로, Redis가 실행되지 않기 때문에 (우리는 Redis 자체가 아닌 Python 라이브러리 만 설치했기 때문에) 여기에서 사용하려는 시도가 실패하고 오류 메시지가 생성 될 것으로 예상해야 합니다.

Note: container 내부에서 host 이름에 액세스하면 실행중인 실행 파일의 프로세스 ID와 같은 컨테이너 ID가 검색됩니다.

이게 다 입니다. 당신은 당신의 시스템에 Python 이나 requirements.txt 에 있는 어떤 것도 필요하지 않으며, 이 images 를 빌드하거나 실행하지 않아도 시스템에 설치됩니다. python 과 flask 로 환경을 설정한 것처럼 보이지는 않습니다.

Build the app

우리는 앱을 만들 준비가 되었습니다. 새 디렉토리의 최상위 레벨에 있는지 확인하십시오. ls:

$ ls
Dockerfile  app.py  requirements.txt

이제 빌드 명령을 실행하십시오. 이렇게 하면 Docker images 가 생성됩니다. 이 이미지는 -t 를 사용하여 tag를 지정하므로 friendlyhello 란 이름을 갖습니다.

Run the app

응용 프로그램을 실행하여 -p를 사용하여 컴퓨터의 포트 4000을 컨테이너의 게시 된 포트 80에 매핑합니다:

$ docker run -p 4000:80 friendlyhello

Python 이 http://0.0.0.0:80 에서 앱을 제공한다는 알림을 보아야합니다. 하지만 그 메시지는 컨테이너 내부에서 들어 오는데, 컨테이너의 포트 80을 4000으로 매핑했는지 모르므로 올바른 URL http://localhost:4000 을 작성합니다.

웹 브라우저에서 해당 URL로 이동하여 "Hello World"텍스트, 컨테이너 ID 및 Redis 오류 메시지를 포함하여 웹 페이지에 표시되는 디스플레이 컨텐트를 확인하십시오.

(이미지 생략)

셸에서 curl 명령을 사용하여 동일한 내용을 볼 수도 있습니다.

$ curl http://localhost:4000
<h3>Hello World!</h3><b>Hostname:</b> 5b2eee1605c4<br/><b>Visits:</b> <i>cannot connect to Redis, counter disabled</i>

Note: 4000:80의 이 포트 재 매핑은 Dockerfile 내에서 EXPOSE하는 것과 Docker run -p 를 사용하여 게시하는 것의 차이를 보여줍니다. 나중 단계에서는 호스트의 포트 80을 컨테이너의 포트 80에 매핑하고 http://localhost 를 사용합니다.

CTRL + C를 눌러 종료하십시오.

이제 백그라운드에서 분리 모드로 앱을 실행해 봅시다.

docker run -d -p 4000:80 friendlyhello

앱에 대한 긴 컨테이너 ID를 얻은 다음 터미널로 다시 가져옵니다. container 가 백그라운드에서 실행 중입니다. docker container ls 로 축약된 container ID를 볼 수도 있습니다 (둘 다 명령을 실행할 때 상호 교환 적으로 작동합니다).

$ docker container ls
# CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS                  NAMES
# 7e782fef4e4b        friendlyhello       "python app.py"     2 minutes ago       Up 2 minutes        0.0.0.0:4000->80/tcp   clever_morse

CONTAINER ID가 http://localhost:4000 에 있는 것과 일치하는 것을 볼 수 있습니다.

다음과 같이 CONTAINER ID를 사용하여 프로세스를 종료하려면 docker container stop을 사용하십시오.

$ docker container stop 7e782fef4e4b

Share your image

방금 만든 이미지의 이식성을 보여주기 위해 빌드 된 이미지를 업로드하고 다른 곳으로 옮기자.

결국 production 에 container 를 배포하려는 경우 레지스트리로 푸시하는 방법을 배워야합니다.

레지스트리는 저장소의 모음이고 저장소는 이미 만들어진 코드를 제외하고는 GitHub 저장소와 같은 종류의 이미지 모음입니다.

레지스트리의 계정은 많은 저장소를 생성 할 수 있습니다.

docker CLI는 기본적으로 Docker의 공용 레지스트리를 사용합니다.

Note: Docker의 공용 레지스트리는 무료이며 사전 구성되어 있기 때문에 여기에서 사용하겠지만, 선택할 수있는 공용 레지스트리가 많으며 Docker Trusted Registry를 사용하여 개인 레지스트리를 설정할 수도 있습니다.

Log in with your Docker ID

Docker 계정이없는 경우 cloud.docker.com 에서 가입하십시오. 사용자 이름을 기록하십시오. 로컬 시스템의 Docker 공용 레지스트리에 로그인하십시오.

$ docker login

Tag the image

로컬 이미지를 레지스트리의 저장소와 연관시키는 표기법은 username/repository:tag 입니다. 이 태그는 선택사항이지만 Docker 이미지에 버전을 제공하는 데 사용하는 메커니즘이기 때문에 권장됩니다. 저장소에 제공하고 get-start:part2 와 같이 context에 의미있는 이름을 tag하십시오. 그러면 이미지가 시작 저장소에 저장되고 part2 로 tag가 지정됩니다.

이제 이미지를 모두 태그를 추가하십시오. 이미지가 원하는 대상에 업로드되도록 사용자 이름, 저장소 및 태그 이름으로 도커 태그 이미지를 실행하십시오. 명령 구문은 다음과 같습니다:

$ docker tag image username/repository:tag

예:

$ docker tag friendlyhello portal2312/get-started:part2

확인:

# $ docker image ls
$ docker images

# REPOSITORY               TAG                 IMAGE ID            CREATED             SIZE
# friendlyhello            latest              2e5340b0b6cf        About an hour ago   167MB
# portal2312/get-started   part2               2e5340b0b6cf        About an hour ago   167MB
# python                   3.6-slim            c23cb4208584        6 days ago          156MB
# hello-world              latest              05a3bd381fc2        6 weeks ago         1.84kB

Publish the image

태그가 지정된 이미지를 저장소에 업로드하십시오:

$ docker push username/repository:tag

Pull and run the image from the remote repository

이제부터는 다음 명령으로 모든 컴퓨터에서 docker run 과 앱을 실행할 수 있습니다:

$ docker run -p 4000:80 username/repository:tag

이미지가 컴퓨터에서 로컬로 사용할 수없는 경우 Docker는 리포지토리에서 이미지를 가져옵니다.

$ docker run -p 4000:80 portal2312/get-started:part2
#

Note: 이 명령의 :tag 부분을 지정하지 않으면 빌드 할 때와 이미지를 실행할 때 모두 :latest 의 태그가 사용됩니다. Docker는 지정된 태그없이 실행 된 이미지의 마지막 버전을 사용합니다 (최신 이미지일 필요는 없습니다).

위치에 관계없이 docker run은 Python 및 requirements.txt 의 모든 종속성과 함께 이미지를 가져 와서 코드를 실행합니다. 모든 것이 깔끔하고 작은 패키지로 이루어지며, 호스트 컴퓨터는 Docker를 설치만 하면 됩니다.

Conclusion of part 2

명령어 모음:

$ docker build -t friendlyname .  # Create image using this directory's Dockerfile
$ docker run -p 4000:80 friendlyname  # Run "friendlyname" mapping port 4000 to 80
$ docker run -d -p 4000:80 friendlyname         # Same thing, but in detached mode
$ docker container ls                                # List all running containers
$ docker container ls -a             # List all containers, even those not running
$ docker container stop <hash>           # Gracefully stop the specified container
$ docker container kill <hash>         # Force shutdown of the specified container
$ docker container rm <hash>        # Remove specified container from this machine
$ docker container rm $(docker container ls -a -q)         # Remove all containers
$ docker image ls -a                             # List all images on this machine
$ docker image rm <image id>            # Remove specified image from this machine
$ docker image rm $(docker image ls -a -q)   # Remove all images from this machine
$ docker login             # Log in this CLI session using your Docker credentials
$ docker tag <image> username/repository:tag  # Tag <image> for upload to registry
$ docker push username/repository:tag            # Upload tagged image to registry
$ docker run username/repository:tag                   # Run image from a registry

Dockerfile

빈 디렉토리를 만듭니다. 디렉토리 (cd)를 새 디렉토리로 변경하고 Dockerfile 이라는 파일을 작성한 후 다음 내용을 해당 파일에 복사하여 붙여넣고 저장하십시오. 새로운 Dockerfile의 각 문장을 설명하는 주석을 기록하십시오.

# Use an official Python runtime as a parent image
FROM python:2.7-slim

# Set the working directory to /app
WORKDIR /app

# Copy the current directory contents into the container at /app
ADD . /app

# Install any needed packages specified in requirements.txt
RUN pip install -r requirements.txt

# Make port 80 available to the world outside this container
EXPOSE 80

# Define environment variable
ENV NAME World

# Run app.py when the container launches
CMD ["python", "app.py"]

프록시 서버를 사용하고 있습니까?

프록시 서버는 일단 실행되면 웹 응용 프로그램에 대한 연결을 차단할 수 있습니다. 프록시 서버를 사용하는 경우 ENV 명령을 사용하여 프록시 서버의 호스트와 포트를 지정하여 다음 행을 Dockerfile에 추가하십시오.

# Set proxy server, replace host:port with values for your servers
ENV http_proxy host:port
ENV https_proxy host:port

이 Dockerfile은 아직 작성하지 않은 몇 가지 파일, 즉 app.py 및 requirements.txt를 참조합니다. 다음 단계를 만들어 보겠습니다.

The app itself

requirements.txt 와 app.py 파일을 두 개 더 만들고 Dockerfile 과 같은 폴더에 두십시오. 이것은 우리의 앱을 완성 시켰는데, 당신이 알듯이 아주 간단합니다. 위의 Dockerfile 이 이미지에 내장되면 Dockerfile의 ADD 명령으로 인해 app.py 및 requirements.txt 가 표시되고 EXPOSE 명령으로 인해 HTTP 를 통해 app.py 의 출력에 액세스 할 수 있습니다.

requirements.txt

Flask
Redis

app.py

from flask import Flask
from redis import Redis, RedisError
import os
import socket

# Connect to Redis
redis = Redis(host="redis", db=0, socket_connect_timeout=2, socket_timeout=2)

app = Flask(__name__)

@app.route("/")
def hello():
    try:
        visits = redis.incr("counter")
    except RedisError:
        visits = "<i>cannot connect to Redis, counter disabled</i>"

    html = "<h3>Hello {name}!</h3>" \
           "<b>Hostname:</b> {hostname}<br/>" \
           "<b>Visits:</b> {visits}"
    return html.format(name=os.getenv("NAME", "world"), hostname=socket.gethostname(), visits=visits)

if __name__ == "__main__":
    app.run(host='0.0.0.0', port=80)

이제 pip install -r requirements.txt 가 Python 용 Flask 및 Redis 라이브러리를 설치하고 app 이 socket.gethostname() 에 대한 호출 결과와 함께 환경 변수 NAME을 인쇄하는지 확인합니다.

마지막으로, Redis가 실행되지 않기 때문에 (우리는 Redis 자체가 아닌 Python 라이브러리 만 설치했기 때문에) 여기에서 사용하려는 시도가 실패하고 오류 메시지가 생성 될 것으로 예상해야 합니다.

Note: container 내부에서 host 이름에 액세스하면 실행중인 실행 파일의 프로세스 ID와 같은 컨테이너 ID가 검색됩니다.

이게 다 입니다. 당신은 당신의 시스템에 Python 이나 requirements.txt 에 있는 어떤 것도 필요하지 않으며, 이 images 를 빌드하거나 실행하지 않아도 시스템에 설치됩니다. python 과 flask 로 환경을 설정한 것처럼 보이지는 않습니다.

Run the app

응용 프로그램을 실행하여 -p를 사용하여 컴퓨터의 포트 4000을 컨테이너의 게시 된 포트 80에 매핑합니다:

$ docker run -p 4000:80 friendlyhello

Python 이 http://0.0.0.0:80 에서 앱을 제공한다는 알림을 보아야합니다. 하지만 그 메시지는 컨테이너 내부에서 들어 오는데, 컨테이너의 포트 80을 4000으로 매핑했는지 모르므로 올바른 URL http://localhost:4000 을 작성합니다.

웹 브라우저에서 해당 URL로 이동하여 "Hello World"텍스트, 컨테이너 ID 및 Redis 오류 메시지를 포함하여 웹 페이지에 표시되는 디스플레이 컨텐트를 확인하십시오.

(이미지 생략)

셸에서 curl 명령을 사용하여 동일한 내용을 볼 수도 있습니다.

$ curl http://localhost:4000
<h3>Hello World!</h3><b>Hostname:</b> 5b2eee1605c4<br/><b>Visits:</b> <i>cannot connect to Redis, counter disabled</i>

Note: 4000:80의 이 포트 재 매핑은 Dockerfile 내에서 EXPOSE하는 것과 Docker run -p 를 사용하여 게시하는 것의 차이를 보여줍니다. 나중 단계에서는 호스트의 포트 80을 컨테이너의 포트 80에 매핑하고 http://localhost 를 사용합니다.

CTRL + C를 눌러 종료하십시오.

이제 백그라운드에서 분리 모드로 앱을 실행해 봅시다.

docker run -d -p 4000:80 friendlyhello

앱에 대한 긴 컨테이너 ID를 얻은 다음 터미널로 다시 가져옵니다. container 가 백그라운드에서 실행 중입니다. docker container ls 로 축약된 container ID를 볼 수도 있습니다 (둘 다 명령을 실행할 때 상호 교환 적으로 작동합니다).

$ docker container ls

CONTAINER ID가 http://localhost:4000 에 있는 것과 일치하는 것을 볼 수 있습니다.

다음과 같이 CONTAINER ID를 사용하여 프로세스를 종료하려면 docker container stop을 사용하십시오.

$ docker container stop 7e782fef4e4b

Share your image

방금 만든 이미지의 이식성을 보여주기 위해 빌드 된 이미지를 업로드하고 다른 곳으로 옮기자.

결국 production 에 container 를 배포하려는 경우 레지스트리로 푸시하는 방법을 배워야합니다.

레지스트리는 저장소의 모음이고 저장소는 이미 만들어진 코드를 제외하고는 GitHub 저장소와 같은 종류의 이미지 모음입니다.

레지스트리의 계정은 많은 저장소를 생성 할 수 있습니다.

docker CLI는 기본적으로 Docker의 공용 레지스트리를 사용합니다.

Note: Docker의 공용 레지스트리는 무료이며 사전 구성되어 있기 때문에 여기에서 사용하겠지만, 선택할 수있는 공용 레지스트리가 많으며 Docker Trusted Registry를 사용하여 개인 레지스트리를 설정할 수도 있습니다.

Log in with your Docker ID

Docker 계정이없는 경우 cloud.docker.com 에서 가입하십시오. 사용자 이름을 기록하십시오. 로컬 시스템의 Docker 공용 레지스트리에 로그인하십시오.

$ docker login

Tag the image

로컬 이미지를 레지스트리의 저장소와 연관시키는 표기법은 username/repository:tag 입니다. 이 태그는 선택사항이지만 Docker 이미지에 버전을 제공하는 데 사용하는 메커니즘이기 때문에 권장됩니다. 저장소에 제공하고 get-start:part2 와 같이 context에 의미있는 이름을 tag하십시오. 그러면 이미지가 시작 저장소에 저장되고 part2 로 tag가 지정됩니다.

이제 이미지를 모두 태그를 추가하십시오. 이미지가 원하는 대상에 업로드되도록 사용자 이름, 저장소 및 태그 이름으로 도커 태그 이미지를 실행하십시오. 명령 구문은 다음과 같습니다:

$ docker tag image username/repository:tag

예:

$ docker tag friendlyhello portal2312/get-started:part2

확인:

# $ docker image ls
$ docker images

# REPOSITORY               TAG                 IMAGE ID            CREATED             SIZE
# friendlyhello            latest              2e5340b0b6cf        About an hour ago   167MB
# portal2312/get-started   part2               2e5340b0b6cf        About an hour ago   167MB
# python                   3.6-slim            c23cb4208584        6 days ago          156MB
# hello-world              latest              05a3bd381fc2        6 weeks ago         1.84kB

Publish the image

태그가 지정된 이미지를 저장소에 업로드하십시오:

$ docker push username/repository:tag

Pull and run the image from the remote repository

이제부터는 다음 명령으로 모든 컴퓨터에서 docker run 과 앱을 실행할 수 있습니다:

$ docker run -p 4000:80 username/repository:tag

이미지가 컴퓨터에서 로컬로 사용할 수없는 경우 Docker는 리포지토리에서 이미지를 가져옵니다.

$ docker run -p 4000:80 portal2312/get-started:part2
#

Note: 이 명령의 :tag 부분을 지정하지 않으면 빌드 할 때와 이미지를 실행할 때 모두 :latest 의 태그가 사용됩니다. Docker는 지정된 태그없이 실행 된 이미지의 마지막 버전을 사용합니다 (최신 이미지일 필요는 없습니다).

위치에 관계없이 docker run은 Python 및 requirements.txt 의 모든 종속성과 함께 이미지를 가져 와서 코드를 실행합니다. 모든 것이 깔끔하고 작은 패키지로 이루어지며, 호스트 컴퓨터는 Docker를 설치만 하면 됩니다.

명령어 모음:

$ docker build -t friendlyname .  # Create image using this directory's Dockerfile
$ docker run -p 4000:80 friendlyname  # Run "friendlyname" mapping port 4000 to 80
$ docker run -d -p 4000:80 friendlyname         # Same thing, but in detached mode
$ docker container ls                                # List all running containers
$ docker container ls -a             # List all containers, even those not running
$ docker container stop <hash>           # Gracefully stop the specified container
$ docker container kill <hash>         # Force shutdown of the specified container
$ docker container rm <hash>        # Remove specified container from this machine
$ docker container rm $(docker container ls -a -q)         # Remove all containers
$ docker image ls -a                             # List all images on this machine
$ docker image rm <image id>            # Remove specified image from this machine
$ docker image rm $(docker image ls -a -q)   # Remove all images from this machine
$ docker login             # Log in this CLI session using your Docker credentials
$ docker tag <image> username/repository:tag  # Tag <image> for upload to registry
$ docker push username/repository:tag            # Upload tagged image to registry
$ docker run username/repository:tag                   # Run image from a registry
