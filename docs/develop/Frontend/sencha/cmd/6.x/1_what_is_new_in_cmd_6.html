새로운 기능에 대한 소개

app init

app init은 현재 Dir를 Sencha Cmd application 으로 초기화합니다.

완전히 비어있는 Dir 에서 시작하여 다음 명령 중 하나를 실행하고

완전히 구조화 된 Ext JS/Cmd "hello world" application 으로 끝낼 수 있습니다.

$ sencha app init --ext=/path/to/extjs/ MyApp --modern

$ sencha app init --ext=/path/to/extjs/ MyApp --classic

$ sencha app init --ext=/path/to/extjs/ MyApp --universal

Note: application은 다음 app build 전까지 완료되지 않을 것입니다.

$ sencha app build

app install

app install은 불완전한 Sencha Cmd application이 실행 가능한 상태에 있음을 보장함.

예를 들어 아주 간단한 application 폴더를 컴퓨터에 놓고 이 명령을 실행하여 전체 Cmd 인프라를 만들 수 있습니다.

예를 들어, 이 github repo를 보고 이를 적용 할 응용 프로그램의 유형을 볼 수 있습니다.

$ sencha app install --frameworks=/path/to/extjs

"frameworks"는 Cmd 6.1에서 소개된 기능. (아래 참조) 여러 버전의 ExtJS가 포함된 폴더를 가리킬 수 있습니다. 위의 repo에서 볼 수 있듯이 ExtJS 버전은 workspace.json 내에 있습니다. frameworks flag는 사용자가 가리키는 폴더 내에서 가능한 한 필요한 버전과 가장 일치하는 항목을 찾으려고 시도합니다.

Note: app build 전까지 완료되지 않을 것입니다.

$ sencha app build

workspace init

현재 Dir가 Sencha Cmd 작업 공간의 일부분인지 확인함.

$ sencha workspace init

# Sencha Cmd v6.2.1.29
# [INF] Using existing workspace at /home/mky/myext

workspace install

불완전한 Sencha Cmd 작업 공간이 실행 ​​가능 상태에 있음을 보장함.

예로, 매우 간단한 작업 공간 Dir를 컴퓨터에 놓고 이 명령을 실행하여 전체 Cmd 인프라를 생성 할 수 있음.

$ sencha workspace install --frameworks=/path/to/extjs

"frameworks"는 Cmd 6.1에서 소개된 기능. (생략)

workspace cleanup command

작업 영역에 더 이상 존재하지 않는 application을 workspace.json의 app 항목에서 제거.

$ sencha workspace cleanup

What's New in Sencha Cmd 6.1

Framework and Workspace Management

Projects가 하나의 작업 공간에서 여러 application을 작업해야 하는 경우에는 다른 framework version이 현실화 될 수 있음.

Sencha Cmd 6.1 이전 버전에서는 모든 application을 동시에 업그레이드해야 하는 ExtJS (및 Sencha Touch 추가 버전) version으로 작업 영역을 구성 할 수 있었음. (불편함)

Sencha Cmd 6.1 부터 workspace.json 파일에 여러 framework를 기술 가능.

# workspace.json

"frameworks": {
    "ext62": "ext",
    "ext602": "ext-6.0.2"
}

frameworks 속성 아래의 각 항목은 현재 작업 공간 내부의 Dir에 해당하며 속성 이름은 app.json 에서 사용할 framework key 입니다.

app.json의 framework 속성은 다음 항목 중 하나에 매핑됨.

# app.json

"framework": "ext62"

workspace에 new framework 추가:

$ sencha framework add /path/to/ext

이 명령은 버전번호(예: ExtJS 6.2.0 == "ext62" or ExtJS 6.0.2 == "ext602")에 따라 제공된 framework에 적절한 키를 결정함.

framework add framework 항목의 모든 측면을 제어할 수 있는 몇 가지 추가 옵션도 제공. 예:

$ sencha framework add -key extFiveO -source /path/to/ext-5.0.0 -path ext50

위와 같이 입력시 workspace.json의 frameworks 에 자동 추가됨:

# workspace.json

"frameworks": {
    "extFive0": "ext50"
}

또한 Cmd는 /path/to/ext-5.0.0의 중요한 framework 파일을 작업 영역 (-source 및 -path 스위치로 지정)의 ext50 하위 폴더에 복사합니다. framework keys는 "ext"또는 "touch"로 시작해야 합니다.

framework가 추가 된 후 framework keys를 사용하여 "generate app" 명령을 전환하여 application을 생성 할 수 있습니다. 예를 들어 위에 추가 한 "extSixTwo" framework를 사용하여 application을 생성하려면 다음과 같이 generate app 명령을 호출해야 함:

$ sencha generate app -extFiveO AppName path/to/app

frameworks 목록 보기:

$ sencha framework list

framework 삭제:

$ sencha framework remove extFive0

$ sencha framework remove --force extFive0

특정 framework를 사용하여 모든 application을 upgrade 하려면 "framework upgrdae" 명령을 사용.

예를 들어 "ext" 키가 있는 framework를 (경로 /path/to/ext) 최신 framework version으로 upgrade 하려면 다음 명령을 실행:

$ sencha framework upgrade ext /path/to/ext

이렇게 하면 framework 파일을 upgrade 하고 이를 사용하는 각 application에 대해 "sencha app upgrade"를 트리거함.

"sencha app upgrade /path/to/ext" 명령을 사용하여 단일 application upgrade 할 수 있음.

이미 작업 공간의 일부인 framework로 upgrade 할 수도 있음.

$ sencha app upgrade ../ext62

"workspace upgrade" 전체 작업 영역에 대해 application 및 package 내의 Cmd 관련 파일을 upgrade하는 방법을 제공.

$ sencha workspace upgrade

What's New in Sencha Cmd 6

New Installers

Sencha Cmd 를 실행하는데 필요한 JRE(Java Runtime Environment)을 포함하도록 설치 프로그램을 변경.

이 설치 프로그램을 사용하면 Java를 별도 다운로드하여 설치할 필요가 없음.

Ruby 종속성 제거, Sencha Cmd는 운영 체제 이상의 전제 조건이 없음을 의미.

우리는 여전히 JRE가 없는 설치 프로그램을 제공할 것.

Non-admin

(Windows, Linux, Mac) non-admin.

No Ruby Dependency

Fashion(아래 참조)이 도입, Ruby 종속성 없음.

  • Live Update.

  • Compass 여전히 사용가능.

Compass (for order Frameworks)

구형 프레임 워크 (Ext JS 4와 5 또는 Sencha Touch 2)에서 Sencha Cmd 6를 사용한다면, Sass 코드를 컴파일하기 위해서는 여전히 Ruby가 필요. 설치 프로그램은 이러한 framework를 지원하는 Compass(및 Sass)의 필수 버전을 설치하도록 지시받을 수 있습니다. 이 설정은 기본적으로 비활성화, 설치 프로그램의 다음 대화 상자에서 활성화할 수 있음.

Fashion

Sench에서 파생 된 Sencha의 새로운 compiler 인 "Fashion"이 포함.

Fashion을 사용하여 ExtJS 6 application 및 테마에 대한 style 규칙(.scss 파일) 작성.

Fashion에서 sencha app watch 사용할 때 *Live Update 를 사용하여 page reload 않고도 application CSS를 동기화 상태로 유지.

자세한 내용은 Fashion Guide 참조.

Workspace Improvement

Package Directories

code sharing with packages를 용이하게 하기 위해 package의 분할 폴더 위치가 도입.

이렇게 하면 생성 한 패키지를 한 위치로 이동하고 다른 위치로 이동하기 위해 필요한 패키지(download 및 추출 된 패키지)를 사용가능.

/                   # The workspace root folder
    packages/       # The former location for all packages
        local/      # New location for generated packages
        remote/     # New location for downloaded/extracted packages

이제 workspace.json 에서 패키지 위치를 구성 할 수 있음.

"/packages/remote" Folder는 source control(.gitignore)에서 "ignored"로 구성 될 수 있음.

이는 해당 폴더의 모든 패키지가 패키지 저장소에서 다운로드되어 다시 필요할 수 있기 때문.

workspace.json

Sencha Cmd 6 사용하면 workspace.json 파일이 작업 영역 루트 폴더에 추가됨.

이 파일은 향후 tooling에서 유용할 것이므로 source control에서 추적해야 함.

우선, 앞서 언급 한 package 경로를 구성 할 수있게 해줌.

Microloader

Sencha Cmd 6는 Microloader의 주요 업데이트를 제공.

Microloader는 Sencha Touch에서 처음 소개 한 응용 프로그램 loader이며

이후 Sencha Cmd 5에서 Ext JS 5 응용 프로그램 용으로 업데이트되었음.

Sencha Cmd 6을 사용하면 최신 Microloader를 Ext JS 6 applications에서 사용할 수 있으며 localStorage Caching을 지원.

이는 Sencha Touch의 Microloader가 작동하는 것과 유사하지만 몇 가지 개선 사항이 있습니다.

  • app.json에서 Caching을 비활성화 할 수 있음

  • microloader content만 microloader에 의해 지워질 것

  • App의 현재 version만 local 저장소에 유지

참조: Microloader Guide

이는 Sencha Touch의 Microloader가 작동하는 것과 유사하지만 몇 가지 개선 사항이 있습니다.

  • app.json에서 Caching을 비활성화 할 수 있음

  • microloader content만 microloader에 의해 지워질 것

  • App의 현재 version만 local 저장소에 유지
