Sencha Cmd는 응용 프로그램의 전체 수명주기에 대해 새 프로젝트를 생성하고 응용 프로그램을 프로덕션 환경으로 배포하는 등 여러 자동화 된 작업을 제공하는 교차 플랫폼 명령 줄 도구입니다.

Meet Sencha Cmd

Sencha Cmd는 Sencha ExtJS 및 Sencha Touch framework와 함께 작동.

강력한 시간 절약 기능 모음을 제공.

Sencha Cmd는 다음과 같은 기능을 제공:

  • Code Generation Tools:

전체 application 생성하고 새로운 MVC 구성 요소로 application을 확장.

  • JS Compiler:

Sencha framework의 의미를 이해하고 소스에서 최소한의 설치 공간을 생성 할 수있는 framework 인식 JavaScript 컴파일러임.

Compiler는 Sencha framework가 제공하는 많은 고급 의미 체계를 최적화하여 application의 load 시간을 줄임.

  • Web Server:

localhost의 파일을 처리하는 경량 웹 서버를 제공.

  • Package Management System:

다른 사용자 나 Sencha package 저장소에서 생성한 package(예: ExtJS Themes)를 쉽게 통합할 수 있는 분산 패키지 관리 시스템.

  • Sencha Web Application Manager Integration

- 서버에 새로운 application version을 쉽게 게시.

  • Workspace Management:

여러 application에 framework, package 및 사용자 정의 코드를 공유하도록 지원.

  • Build Script:

"이전" 및 "이후" 확장 점이 있는 application 및 packages 용 biuld script를 생성하여

특정 요구 사항에 맞게 build process를 사용자 정의 할 수 있음.

  • Cordova/PhoneGap Integration:

웹 애플리케이션을 일등급, 장치 기능에 액세스할 수 있고

App Store에 배포 할 수있는 mobile application.

  • Image Capture:

CSS3 기능을 레거시 브라우저용(예: 테두리 반경 및 선형 그라데이션) 스프라이트로 변환.

  • Tuning Tools:

강력한 코드 선택 도구는 응용 프로그램의 최종 빌드에 포함 된 것을 조정하고,

페이지 전체의 공통 코드를 결정하고,

고급 설정 작업을 사용하여 공유 코드를 작성하므로 원하는대로 정확하게 빌드 할 수 있음.

  • Flexible Configuration System:

application 또는 작업 영역 레벨 또는 시스템의 모든 작업 영역에서 명령 옵션에 기본값을 지정.

  • Logging:

명령의 내부 동작을 이해하고 문제 해결을 용이하게하는 강력한 logging.

  • Third-party Software:

Sencha Touch 및 ExtJS 5 이전버전의 Sencha Cmd에는 Compass와 Sass의 호환 버전이 포함되어 있음.

  • Code Generation Hooks:

한 페이지에만 한정되거나 작업 공간의 모든 페이지에서 공유 할 수 있습니다.

(예: 새 model 생성시 코딩 규칙 또는 지침을 확인)

Compatibility (적합성)

Sencha Cmd는 Sencha Ext JS 버전 4.1.1a 이상 및 Sencha Touch 버전 2.1 이상을 지원합니다. Sencha Cmd의 많은 기능에는 이러한 버전 또는 이후 버전에서만 사용할 수있는 프레임 워크 지원이 필요합니다. 일부 저수준 명령은 이전 버전의 Sencha 프레임 워크 또는 일반적으로 JavaScript에서 사용할 수 있습니다.

이전 버전의 Ext JS를 사용하는 경우 Sencha Cmd의 빌드 명령을 사용하여 JSB 파일을 통해 빌드 할 수 있습니다. 즉, Sencha Cmd는 JSBuilder를 대체하여 JSB 파일에 설명 된 파일의 압축 된 빌드를 생성 할 수 있습니다. Sencha Cmd는 더 이상 사용되지 않는 SDK 도구 v2에서 수행 한대로 JSB 파일을 업데이트하지 않습니다.

Sencha Touch 2.0과 Sencha Ext JS 4.0은 추후 Touch 나 Ext JS 버전에서는 사용할 수없는 SDK 도구 v2가 필요합니다.

Upgrading Sencha Cmd

sencha upgrade Sencha cmd를 upgrade 함

새로운 update 확인:

$ sencha upgrade --check

# Sencha Cmd v6.2.1.29
# [INF] Determining the latest version of Sencha Cmd
# [INF] Downloading : ....................
# [INF] The latest version of Sencha Cmd is 6.2.1.29
# [INF] Sencha Cmd 6.2.1.29 is this version.

--check 옵션을 지정하지 않으면, sencha upgrade 명령은 최신 version 을 갖고있지 않은 경우다운로드하고 설치함.

설치가 끝나면 새로운 콘솔이나 터미널을 시작하여 PATH 환경 변수의 변경 사항을 가져옴.

Sencha Cmd의 여러 버전을 나란히 설치할 수 있으므로 새 릴리스를 안전하게 제거하고 PATH 또는 심볼 링크를 제거하여 이전 버전으로 돌아갈 수 있음. 그러나 Sencha app upgrade를 사용하여 applications을 업그레이드하는 것은 소스 제어 항목에 더 가깝습니다. 이전 Sencha Cmd로 downgrade할 경우 "roll back" 해야 할 수도 있음.

Current Directory

대부분의 경우, Sencha Cmd는 특정 현재 디렉토리를 설정해야 함.

또는 관련 Sencha SDK에 대한 세부 정보 만 알면 됨.

SDK (또는 "framework")는 생성된 application 폴더에서 실행될 때 Sencha Cmd에 의해 자동으로 결정되거나 추출 된 SDK 폴더에서 몇 가지 명령에 대해 자동으로 결정될 수 있음.

중요: 다음 명령의 경우 Sencha Cmd는 생성 된 application의 root 폴더에서 실행해야 함. application의 root 폴더에서 실행하지 않으면 명령이 실패함.

* sencha generate ... (for commands other than `app`, `package` and `workspace`)
* sencha app ...

이는 패키지에도 해당됨. sencha package build 같은 명령 실행시 현재 Dir 원하는 package 폴더여야 함.

Troubleshooting

Sencha Cmd를 사용시 발생하는 일반적인 문제를 해결 위한 tip.

Java Heap Space

Cmd 명령을 실행 후 "java heap space" 오류가 발생하면 Java의 할당 된 메모리 사용을 늘려야 함.

Mac & Linux

# ~/.bash_profile

export _JAVA_OPTIONS="-Xms1024m -Xmx2048m"

Windows

변수를 환경 변수로 추가하거나 startup.bat에 다음을 추가 할 수 있음.

# startup.bat

set _JAVA_OPTIONS="-Xms1024m -Xmx2048m"

Command Not Found

sencha를 실행하면 오류 메시지 sencha:OSX/Linux 또는 'sencha'에서 찾을 수없는 명령이 Windows의 내부 또는 외부 명령, 실행 가능 프로그램 또는 배치 파일로 인식되지 않는 경우 다음 단계를 수행하십시오:

  • terminal/command prompt 창을 닫고 새 창을 열기.

  • Sencha Cmd가 제대로 설치되었는지 확인.

    • 설치 디렉토리가 존재합니다. 기본적으로 설치 경로는 다음과 같습니다:
    • Windows: C:\Users\Me\bin\Sencha\Cmd{버전}
    • MacOS X: ~/bin/Sencha/Cmd/{버전}
    • Linux: ~/bin/Sencha/Cmd/{version}
    • Sencha Cmd Dir의 경로는 PATH 환경 변수 앞에 추가됨. 터미널에서 Windows에서 echo % PATH %를 실행하거나 Mac 또는 Linux에서 echo $ PATH를 실행. Sencha Cmd Dir는 출력의 일부에 표시되어야함. 그렇지 않은 경우 수동으로 PATH에 추가하기.

Wrong Current Directory

일반적인 실수는 현재 Dir가 추출 된 SDK Dir 또는 application Dir가 되어야 하지만 해당 Dir가 설정되지 않은 명령을 수행하는 것.

이 요구 사항이 충족되지 않으면 Sencha Cmd가 오류를 표시하고 종료함.

참고: 올바른 application Dir는 Sencha Cmd에서 생성 된 application Dir 임.

Errors While Resolving Dependencies

sencha app build 명령은 index.html 및 app.json 파일을 읽고 필수 클래스를 검색하여 작동함. 만일 application에 필요한 클래스를 제대로 선언하지 않으면 일반적으로 build가 완료되지만 application에 필요한 모든 클래스가 포함되지는 않음. 필수 클래스를 모두 지정하려면 항상 debugger console (IE/Chrome의 '개발자 도구', FireFox의 FireBug 및 Safari의 웹 관리자)을 사용하여 개발하고 나타나는 모든 경고 및 오류 메시지를 해결하십시오.

다음과 같은 경고가 표시 될 때마다:

[Ext.Loader] Synchronously loading 'Ext.foo.Bar'; consider adding 'Ext.foo.Bar'
explicitly as a require of the corresponding class

Dependency가 발생한 클래스의 requires 배열 속성 안에 'Ext.foo.Bar'를 추가해야 함.

application 전반의 종속성인 경우 app.js의 Ext.application(...) 문에서 requires 배열 속성에 추가합니다.

Current Directory

대부분의 경우, Sencha Cmd는 특정 현재 디렉토리를 설정해야 함.

또는 관련 Sencha SDK에 대한 세부 정보 만 알면 됨.

SDK (또는 "framework")는 생성된 application 폴더에서 실행될 때 Sencha Cmd에 의해 자동으로 결정되거나 추출 된 SDK 폴더에서 몇 가지 명령에 대해 자동으로 결정될 수 있음.

중요: 다음 명령의 경우 Sencha Cmd는 생성 된 application의 root 폴더에서 실행해야 함. application의 root 폴더에서 실행하지 않으면 명령이 실패함.

* sencha generate ... (for commands other than `app`, `package` and `workspace`)
* sencha app ...

이는 패키지에도 해당됨. sencha package build 같은 명령 실행시 현재 Dir 원하는 package 폴더여야 함.

Troubleshooting

Sencha Cmd를 사용시 발생하는 일반적인 문제를 해결 위한 tip.

Java Heap Space

Cmd 명령을 실행 후 "java heap space" 오류가 발생하면 Java의 할당 된 메모리 사용을 늘려야 함.

Mac & Linux

# ~/.bash_profile

export _JAVA_OPTIONS="-Xms1024m -Xmx2048m"

Windows

변수를 환경 변수로 추가하거나 startup.bat에 다음을 추가 할 수 있음.

# startup.bat

set _JAVA_OPTIONS="-Xms1024m -Xmx2048m"

Command Not Found

sencha를 실행하면 오류 메시지 sencha:OSX/Linux 또는 'sencha'에서 찾을 수없는 명령이 Windows의 내부 또는 외부 명령, 실행 가능 프로그램 또는 배치 파일로 인식되지 않는 경우 다음 단계를 수행하십시오:

  • terminal/command prompt 창을 닫고 새 창을 열기.

  • Sencha Cmd가 제대로 설치되었는지 확인.

    • 설치 디렉토리가 존재합니다. 기본적으로 설치 경로는 다음과 같습니다:
    • Windows: C:\Users\Me\bin\Sencha\Cmd{버전}
    • MacOS X: ~/bin/Sencha/Cmd/{버전}
    • Linux: ~/bin/Sencha/Cmd/{version}
    • Sencha Cmd Dir의 경로는 PATH 환경 변수 앞에 추가됨. 터미널에서 Windows에서 echo % PATH %를 실행하거나 Mac 또는 Linux에서 echo $ PATH를 실행. Sencha Cmd Dir는 출력의 일부에 표시되어야함. 그렇지 않은 경우 수동으로 PATH에 추가하기.

Wrong Current Directory

일반적인 실수는 현재 Dir가 추출 된 SDK Dir 또는 application Dir가 되어야 하지만 해당 Dir가 설정되지 않은 명령을 수행하는 것.

이 요구 사항이 충족되지 않으면 Sencha Cmd가 오류를 표시하고 종료함.

참고: 올바른 application Dir는 Sencha Cmd에서 생성 된 application Dir 임.

Errors While Resolving Dependencies

sencha app build 명령은 index.html 및 app.json 파일을 읽고 필수 클래스를 검색하여 작동함. 만일 application에 필요한 클래스를 제대로 선언하지 않으면 일반적으로 build가 완료되지만 application에 필요한 모든 클래스가 포함되지는 않음. 필수 클래스를 모두 지정하려면 항상 debugger console (IE/Chrome의 '개발자 도구', FireFox의 FireBug 및 Safari의 웹 관리자)을 사용하여 개발하고 나타나는 모든 경고 및 오류 메시지를 해결하십시오.

다음과 같은 경고가 표시 될 때마다:

[Ext.Loader] Synchronously loading 'Ext.foo.Bar'; consider adding 'Ext.foo.Bar'
explicitly as a require of the corresponding class

Dependency가 발생한 클래스의 requires 배열 속성 안에 'Ext.foo.Bar'를 추가해야 함.

application 전반의 종속성인 경우 app.js의 Ext.application(...) 문에서 requires 배열 속성에 추가합니다.
