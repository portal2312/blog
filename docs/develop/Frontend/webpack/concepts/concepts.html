핵심적으로 webpack 은 최신 JavaScript 응용 프로그램을위한 정적 모듈 번들러 입니다. 웹팩은 애플리케이션을 처리 할 때 프로젝트에 필요한 모든 모듈을 매핑하고 하나 이상의 번들을 생성 하는 종속성 그래프 를 내부적으로 빌드합니다.

JavaScript 모듈 및 웹팩 모듈에 대한 자세한 내용은 여기를 참조하십시오.

버전 4.0.0부터 웹팩에는 프로젝트를 번들하기 위해 구성 파일 이 필요하지 않습니다. 그럼에도 불구하고, 그것은 당신의 요구에 더 잘 맞도록 믿을 수 없을 정도로 구성 가능 합니다.

시작하려면 핵심 개념 만 이해하면 됩니다:

  • Entry (기입)
  • Output (산출)
  • Loaders (불러오기)
  • Plugins (플러그인)
  • Mode (방법)
  • Browser Compatibility (브라우저 호환성)

이 문서는 이러한 개념에 대한 고급 개요를 제공하기 위해 작성되었습니다.
자세한 개념별 사용 사례에 대한 링크를 제공합니다.

모듈 번들러의 기본 개념과 이들이 어떻게 작동하는지 이해하려면 다음 리소스를 참조하십시오:

  • 수동으로 응용 프로그램 번들링
  • 간단한 모듈 번들러 라이브 코딩
  • 간단한 모듈 번들러에 대한 자세한 설명

Entry

entry 은 내부 종속성 그래프 작성을 시작하기 위해 사용해야하는 모듈 웹팩을 나타냅니다. webpack은 진입 점이 의존하는 다른 모듈과 라이브러리를 (직접적으로) 파악합니다.

기본적으로 값은 ./src/index.js 이지만 entry webpack 구성에서 특성을 설정하여 다른 (또는 여러 진입 점)을 지정할 수 있습니다. 예를 들면 다음과 같습니다:

webpack.config.js

module.exports = {
  entry: './path/to/my/entry/file.js'
};

entry points 섹션에서 자세히 알아보십시오 .

Output

output 속성은 웹팩에 번들을 생성 할 위치와 파일 이름을 지정하는 방법을 알려줍니다. 기본 출력 파일의 경우 기본값은 ./dist/main.js이고 다른 생성 된 파일의 경우 ./dist 폴더입니다.

구성에서 output 필드를 지정하여 프로세스의 이 부분을 구성 할 수 있습니다:

webpack.config.js

const path = require('path');

module.exports = {
  entry: './path/to/my/entry/file.js',
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: 'my-first-webpack.bundle.js'
  }
};

위의 예제에서 output.filename 와 output.path 속성을 사용하여 웹팩에 번들 이름과 배포 위치를 알려줍니다. 맨 위에 경로 모듈을 가져 오는 것에 대해 궁금한 경우 파일 경로를 조작하는 데 사용되는 핵심 Node.js 모듈입니다.

이 output 속성에는 더 많은 구성 가능한 기능이 있습니다. 그 뒤에있는 개념에 대해 배우려면 출력 섹션에서 더 많은 내용을 읽을 수 있습니다 .

Loader

기본적으로 webpack 은 JavaScript 및 JSON 파일만 이해합니다. Loader 는 webpack 이 다른 유형의 파일을 처리하고 이를 응용 프로그램이 소비하고 종속성 그래프에 추가 할 수있는 유효한 모듈로 변환 할 수 있도록 합니다.

모든 유형의 모듈을 가져 오는 기능 (예: .css 파일은 웹팩 전용 기능이며 다른 번들러 또는 작업 실행기에서 지원하지 않을 수 있습니다. 개발자는 보다 정확한 의존성 그래프를 작성할 수 있으므로 언어의 확장이 보장된다고 생각합니다.

상위 레벨에서 로더 는 웹팩 구성에 두 가지 특성이 있습니다:

  1. 이 test 특성은 변환 할 파일을 식별합니다.
  2. 이 use 특성은 변환을 수행하는 데 사용해야하는 로더를 나타냅니다.

webpack.config.js

const path = require('path');

module.exports = {
  output: {
    filename: 'my-first-webpack.bundle.js'
  },
  module: {
    rules: [
      { test: /\.txt$/, use: 'raw-loader' }
    ]
  }
};

구성은 위에서 정의한 rules 두 필수 특성 단일 모듈 속성: test 및 use. 이것은 webpack의 컴파일러에게 다음을 알려줍니다:

"Hywebpack 컴파일러는 require()/ import문 안에서 '.txt'파일로 해석되는 경로를 발견 할 때 번들에 추가하기 전에를 사용 하여 raw-loader변환하십시오."

웹팩 구성에서 규칙을 정의 할 때는 규칙을 정의 module.rules하고 그렇지 않은 규칙을 정의해야합니다 rules. 귀하의 이익을 위해, 웹팩은 이것이 잘못 수행되면 경고합니다.

정규식을 사용하여 파일을 일치시킬 때는 인용 할 수 없습니다. 즉 /.txt$/동일하지 않습니다 '/.txt$/'나 "/.txt$/". 전자는 webpack에게 .txt로 끝나는 모든 파일을 일치 시키도록 지시하고 후자는 webpack에게 절대 경로 '.txt'를 갖는 단일 파일을 일치 시키도록 지시합니다. 이것은 당신의 의도가 아닐 것입니다.

loaders section에 loader를 포함시킬 때 추가 사용자 정의를 확인할 수 있습니다.

Plugins

loaders 는 특정 유형의 모듈을 변환하는 데 사용되지만 plugins 은 번들 최적화, 자산 관리 및 환경 변수 주입과 같은 광범위한 작업을 수행하는 데 사용될 수 있습니다.

plugin interface 과 어떻게 이것을 사용하여 웹팩의 기능을 확장하는 방법 확인하십시오.

plugins 을 사용하려면 require()에 plugins배열에 추가해야합니다. 대부분의 플러그인은 옵션을 통해 사용자 정의할 수 있습니다. 다른 목적으로 구성에서 플러그인을 여러 번 사용할 수 있으므로 new 연산자를 사용하여 플러그인을 호출하여 인스턴스를 작성해야 합니다.

webpack.config.js

const HtmlWebpackPlugin = require('html-webpack-plugin'); //installed via npm
const webpack = require('webpack'); //to access built-in plugins

module.exports = {
  module: {
    rules: [
      { test: /\.txt$/, use: 'raw-loader' }
    ]
  },
  plugins: [
    new HtmlWebpackPlugin({template: './src/index.html'})
  ]
};

위의 예제에서 html-webpack-plugin 생성된 모든 번들을 자동으로 삽입하여 애플리케이션의 HTML 파일을 생성합니다.

webpack이 기본적으로 제공하는 많은 플러그인이 있습니다! 플러그인 목록을 확인하십시오.

웹팩 설정에서 플러그인을 사용하는 것은 간단합니다. 그러나 추가로 조사 할 가치가있는 많은 사용 사례가 있습니다. 이에 대한 자세한 내용은 여기를 참조하십시오.

Mode

설정에 따라 mode 하나에 매개 변수를 development, production 또는 none각 환경에 대응 웹팩에 내장 된 최적화를 활성화 할 수 있습니다. 기본값은 production입니다.

module.exports = {
  mode: 'production'
};

여기 에서 모드 구성 및 각 값에서 수행되는 최적화 에 대해 자세히 알아보십시오.

Browser Compatibility

webpack 은 ES5와 호환 되는 모든 브라우저를 지원합니다 (IE8 이하는 지원되지 않음). webpack 은 필요 Promise에 대한 import() 및 require.ensure(). 구형 브라우저를 지원하려면 이러한 표현식을 사용하기 전에 polyfill 을 로드 해야합니다 .
