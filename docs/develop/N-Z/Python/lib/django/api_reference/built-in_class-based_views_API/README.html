Specification

클래스 기반보기에서 제공하는 각 요청은 독립적 인 상태입니다.

따라서 인스턴스에 상태 변수를 저장하는 것이 안전합니다.

(즉, self.foo = 3은 스레드 안전 연산입니다)

클래스 기반 뷰는 as_view()클래스 메소드를 사용하여 URL 패턴에 배포됩니다:

# urls.py

urlpatterns = [
    url(r'^view/$', MyView.as_view(size=42)),
]

NOTE: Thread safety with view arguments (뷰 인수가있는 스레드 안전성)

뷰에 전달 된 인수는 뷰의 모든 인스턴스간에 공유됩니다.
즉, 목록의 사전이나 다른 가변 객체를 뷰의 인수로 사용하면 안됩니다.
공유 객체를 수정하면 공유 뷰를 방문하는 한 사용자의 작업이 동일한 보기를 방문하는
후속 사용자에게 영향을 미칠 수 있습니다.

as_view ()에 전달 된 인수는 요청을 처리하는 데 사용되는 인스턴스에 할당됩니다.

이전 예제를 사용하면 MyView의 모든 요청에서 self.size를 사용할 수 있음을 의미합니다.

인수는 이미 클래스에있는 속성과 일치해야합니다 (hasattr 검사에서 True를 반환).

Base vs Generic views

Base class-based views는 부모 뷰로 생각할 수 있으며, 자체적으로 사용되거나 상속 될 수 있습니다.

그것들은 프로젝트에 필요한 모든 기능을 제공하지 않을 수도 있습니다.

이 경우 Mixins이 Base Views가 할 수있는 것을 확장합니다.

Django의 generic views는 base views를 기반으로 하며 객체의 세부 정보 표시와

같은 일반적인 사용 패턴의 바로 가기로 개발되었습니다.

그들은 views 개발에서 발견되는 특정 공통 관용구 및 패턴을 취해 반복하여 자신을 반복하지 않고도

일반적인 데이터보기를 빠르게 작성할 수 있습니다.

대부분의 generic views 에는 Queryset 인스턴스인 queryset key 가 필요합니다.

QuerySet 객체에 대한 자세한 정보는 쿼리 만들기를 참조하십시오.
