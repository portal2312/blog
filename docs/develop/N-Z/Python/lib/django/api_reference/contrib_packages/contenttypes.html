Generic relations

자신의 모델 중 하나에서 외래키를 ContentType 에 추가하면 모델이 위의 Permission 모델의 예제에서와 같이 다른 모델 클래스에 효과적으로 묶일 수 있습니다.

그러나 한 단계 더 나아가 ContentType 을 사용하여 모델간에 진정으로 일반적인 ("polymorphic"이라고도 함) 관계를 가능하게 할 수 있습니다.

간단한 예제는 태깅 시스템입니다. 다음과 같이 보일 수 있습니다:

from django.contrib.contenttypes.fields import GenericForeignKey
from django.contrib.contenttypes.models import ContentType
from django.db import models

class TaggedItem(models.Model):
    tag = models.SlugField()
    content_type = models.ForeignKey(ContentType, on_delete=models.CASCADE)
    object_id = models.PositiveIntegerField()
    content_object = GenericForeignKey('content_type', 'object_id')

    def __str__(self):
        return self.tag

일반적인 ForeignKey 는 하나의 다른 모델을 "가리킬" 수 있습니다. 즉, TaggedItem 모델이 ForeignKey를 사용하면 태그를 저장할 하나의 모델만 선택해야 합니다. contenttypes 애플리케이션은 이 문제를 해결하고 모든 모델과의 관계를 허용하는 특수 필드 유형 (GenericForeignKey)을 제공합니다.

class GenericForeignKey

GenericForeignKey 를 설정하는 세 부분이 있습니다.

  1. 모델에 ContentType에 대한 ForeignKey를 지정하십시오.

  2. 이 필드의 일반적인 이름은 "content_type"입니다. 모델에 관련 모델의 기본 키 값을 저장할 수있는 필드를 제공하십시오. 대부분의 모델에서 PositiveIntegerField를 의미합니다. 이 필드의 일반적인 이름은 "object_id"입니다.

  3. 모델에 GenericForeignKey를 지정하고 위에서 설명한 두 필드의 이름을 전달하십시오. 이러한 필드의 이름이 "content_type"및 "object_id"인 경우이를 생략 할 수 있습니다. 이는 GenericForeignKey가 찾을 기본 필드 이름입니다.

for_concrete_model

False 인 경우 필드는 프록시 모델을 참조 할 수 있습니다. 기본값은 True입니다. 이것은 for_concrete_model 인수를 get_for_model()과 동일화합니다.

Primary key type compatibility

"object_id"필드는 관련 모델의 기본 키 필드와 동일한 유형 일 필요는 없지만 기본 키 값은 get_db_prep_value () 메소드로 "object_id"필드와 동일한 유형으로 강제 변환 할 수 있어야합니다.

Serializing references to ContentType objects

일반 관계를 구현하는 모델에서 데이터를 직렬화하는 경우 (예: fixtures를 생성 할 때) 자연스러운 키를 사용하여 관련 ContentType 객체를 고유하게 식별해야합니다. 자세한 정보는 자연 키와 dumpdata - natural-foreign를 참조하십시오.

이것에 의해, 통상의 ForeignKey 로 사용되는 API와 같은 API가 사용 가능하게 됩니다. 각 TaggedItem 에는 관련된 객체를 반환하는 content_object 필드가 있으며, TaggedItem 을 만들 때 해당 필드에 할당하거나 사용할 수도 있습니다.

from django.contrib.auth.models import User
guido = User.objects.get(username='Guido')
t = TaggedItem(content_object=guido, tag='bdfl')
t.save()
t.content_object
<User: Guido>

Different ForeignKey

이제 우리는 더 이상 ForeignKey를 투표 및 투표와 같은 선호하는 활동을 추적하려는 다른 모델에 보관하지 않습니다. 우리는 이제 활동 모델을 수정하지 않고도 원하는 모든 모델에 대해 이러한 활동을 추적 할 수 있습니다.

관계는 활동을 추적하려는 모델에서 작성됩니다.

또한 관계에보다 의미있는 이름을 정의 할 수 있습니다. 예를 들어 사용자는 게시 및 댓글 모델과 상호 작용하여이를 좋아할 수 있습니다. Answer 모델을 사용하는 동안 투표 / 투표 만 가능합니다. 마지막으로 Question 모델을 사용하면 사용자가 투표 / 투표를 통해 좋아할 수 있습니다.

좋은 점은 액티비티와 상호 작용하려는 새로운 모델을 가지고 있다면 간단히 GenericRelation을 추가하는 것입니다:

관련 객체가 삭제되면 content_type 및 object_id 필드는 원래 값으로 설정되고 GenericForeignKey는 None을 반환합니다.

또한 GenericRelation이있는 객체를 삭제하면 GenericForeignKey가 가리키는 객체도 삭제됩니다. 위의 예에서 책갈피 개체가 삭제되면이를 가리키는 TaggedItem 개체가 동시에 삭제된다는 것을 의미합니다.

ForeignKey와 달리 GenericForeignKey는 on_delete 인수를 사용하여이 동작을 사용자 정의하지 않습니다.

원하는 경우 GenericRelation을 사용하지 않고 계단식 삭제를 피할 수 있으며 pre_delete 신호를 통해 대체 동작을 제공할 수 있습니다.
