Introducing automated testing

What are autumated tests?

한 번 테스트 세트를 작성한 다음 앱을 변경하면서 시간이 오래 걸리는 수동 테스트를 수행하지 않아도 원래 의도대로 코드가 작동하는지 확인할 수 있습니다.

Why you need to create tests

자동화 된 테스트를 만드는 데 어려움을 겪고 있으면 더 나은 결과를 얻지 못할 것.

Tests will save you time

시간 절약, 의존성 문제 테스트, 매번 또는 즉시 테스트 가능.

Tests don't just identify problems, they prevent them

테스트는 문제를 식별 뿐만 아니라 문제를 방지함.

Tests make your code more attractive

신뢰성 얻을 수 있음.

Tests help teams work together

테스트는 팀과의 협업을 돕는다.

장고 프로그래머로서 생계를 꾸려 나가려면 테스트를 잘해야 함!

Basic testing strategies

  • ""테스트 중심 개발"이라는 원칙을 따름.

  • 그들은 실제로 코드를 작성하기 전에 테스트를 작성.

  • 이것은 반 직관적으로 보일지 모르지만,

실제로는 대부분의 사람들이 자주 수행하는 것과 유사.

문제를 설명하고 문제를 해결하기 위해 코드를 작성합니다.

  • 수천 줄의 파이썬을 작성했다면, 테스트 할 대상을 선택하는 것이 쉽지 않을 수도 있음.

이 경우 다음 번에 변경 사항을 적용 할 때 새로운 기능을 추가하거나 버그를 수정할 때

첫 번째 테스트를 작성하는 것이 효과적.

Writing our first test

We identify a bug

# XXX: Find bug

import datetime
from django.utils import timezone
from polls.models import Question

future_question = Question(pub_date=timezone.now() + datetime.timeelta(days=30))
future_question.was_published_recently()
# True

Create a test to expose the bug

문제를 테스트하기 위해 shell에서 방금 수행 한 작업은 자동화 된 테스트에서 수행할 수 있는 작업이므로 자동화 된 테스트로 바꾸자. 애플리케이션의 테스트를 위한 일반적인 장소는 애플리케이션의 tests.py 파일에 있음. 테스트 시스템은 이름이 test로 시작하는 모든 파일에서 테스트를 자동으로 찾습니다. polls 응용 프로그램의 tests.py 파일에 다음을 입력하십시오.

# ./polls/test.py
import datetime

from django.utils import timezone
from django.test import TestCase

from .models import Question


class QuestionMethodTests(TestCase):

    def test_was_published_recently_with_future_question(self):
        """
        was_published_recently() should return False for questions whose
        pub_date is in the future.
        """
        time = timezone.now() + datetime.timedelta(days=30)
        future_question = Question(pub_date=time)
        self.assertIs(future_question.was_published_recently(), False)

여기에서 수행 한 작업은 pub_date 를 사용하여 Question 인스턴스를 생성하는 method로 django.test.TestCase 서브 클래스를 작성하는 것입니다. 그런 다음 was_published_recently() 의 결과를 확인합니다. 이 값은 False 여야합니다.

테스트 행위:

  • python manage.py test polls 는 polls Apps의 tests를 바라봄.

  • django.test.TestCase class 의 subclass 를 찾음.

  • 특별한 데이터베이스를 테스트 목적으로 생성함.

  • test methods 가 바라보았음 - test 로 시작하는 이름인 것

  • test_was_published_recently_with_future_question 에서 pub_date field가 미래에 30일인 Question 인스턴스를 생성.

  • ... assertIs() method 사용

    was_published_recently()가 결과가 True를 알았지만 False를 반환하기를 원함.

테스트는 어떤 테스트가 실패했는지와 실패가 발생한 행까지 알려줌.

Fixing the bug

# ./polls/models.py
# ...
def was_published_recently(self):
    now = timezone.now()
    return now - datetime.timedelta(days=1) <= self.pub_date <= now

More comprehensive tests

하나의 method를 테스트하기 위해 Test method를 여러개 추가해야되는 당혹스러움.

# ./polls/tests.py

def test_was_published_recently_with_old_question(self):
    """
    was_published_recently() should return False for questions whose
    pub_date is older than 1 day.
    """
    time = timezone.now() - datetime.timedelta(days=30)
    old_question = Question(pub_date=time)
    self.assertIs(old_question.was_published_recently(), False)

def test_was_published_recently_with_recent_question(self):
    """
    was_published_recently() should return True for questions whose
    pub_date is within the last day.
    """
    time = timezone.now() - datetime.timedelta(hours=1)
    recent_question = Question(pub_date=time)
    self.assertIs(recent_question.was_published_recently(), True)

간단한 apps 라도 앞으로 더 복잡해질 것이고 상호 작용하는 다른 코드가 무엇이든 간에 테스트를 작성한 메소드가 예상대로 작동 할 것이라는 보장이 있습니다.

(한번 작성해 놓으면 많이 테스트 코드를 추가해야 되지만 다시 테스트 할때는 빠르게 확인 가능)

Test a view

A test for a view

먼저 테스트를 작성한 다음 수정하기위한 코드를 작성했음. 사실 테스트 중심 개발의 간단한 예 였지만 실제로 어떤 순서로 작업든 중요하지 않음.

첫 번째 테스트에서는 코드의 내부 동작을 자세히 살펴 보았습니다. 이 테스트에서는 웹 브라우저를 통해 사용자가 경험할 수있는 동작을 확인하려고합니다.

문제를 해결하기 전에 도구를 살펴 보겠습니다.

The Django test client

Django는 test client 를 제공하여 뷰 레벨에서 코드와 상호 작용하는 사용자를 시뮬레이트 가능.

tests.py 또는 shell 에서도 사용가능.

$ python manage.py shell

shell 에서 test 환경을 설정.

from django.test.utils import setup_test_environment
setup_test_environment()

setup_test_environment() : template renderer 설치.

template renderer : response.context와 같은 응답에 대한 추가 속성을 검사 가능.

주의

  • 이 방법은 test Database를 설정하지 않으므로 기존 Database에 대해 다음이 실행되며 이미 작성한 질문에 따라 결과가 약간 다를 수 있음.

  • settings.py 의 TIME_ZONE 올바르지 않은 경우 예기치 않은 결과가 발생할 수 있음.

  • 초기 설정을 기억하지 못하면 계속하기 전에 먼저 확인할 것.

Testing our new view

이제는 runserver를 실행하고 브라우저에 사이트를 로드하고 과거와 미래의 날짜가 포함 된 질문을 작성하고 게시 된 것만 나열되는지 확인하여 예상대로 작동합니다. 이 작업에 영향을 줄 수있는 변경을 할 때마다 매번 그렇게하지 않으려 고합니다. 위의 쉘 세션을 기반으로 테스트를 만들어 보기.

# ./polls/tests.py
from django.urls import reverse
# ...

def create_question(question_text, days):
    """
    Creates a question with the given `question_text` and published the
    given number of `days` offset to now (negative for questions published
    in the past, positive for questions that have yet to be published).
    """
    time = timezone.now() + datetime.timedelta(days=days)
    return Question.objects.create(question_text=question_text, pub_date=time)


class QuestionViewTests(TestCase):
    def test_index_view_with_no_questions(self):
        """
        If no questions exist, an appropriate message should be displayed.
        """
        response = self.client.get(reverse('polls:index'))
        self.assertEqual(response.status_code, 200)
        self.assertContains(response, "No polls are available.")
        self.assertQuerysetEqual(response.context['latest_question_list'], [])

    def test_index_view_with_a_past_question(self):
        """
        Questions with a pub_date in the past should be displayed on the
        index page.
        """
        create_question(question_text="Past question.", days=-30)
        response = self.client.get(reverse('polls:index'))
        self.assertQuerysetEqual(
            response.context['latest_question_list'],
            ['<Question: Past question.>']
        )

    def test_index_view_with_a_future_question(self):
        """
        Questions with a pub_date in the future should not be displayed on
        the index page.
        """
        create_question(question_text="Future question.", days=30)
        response = self.client.get(reverse('polls:index'))
        self.assertContains(response, "No polls are available.")
        self.assertQuerysetEqual(response.context['latest_question_list'], [])

    def test_index_view_with_future_question_and_past_question(self):
        """
        Even if both past and future questions exist, only past questions
        should be displayed.
        """
        create_question(question_text="Past question.", days=-30)
        create_question(question_text="Future question.", days=30)
        response = self.client.get(reverse('polls:index'))
        self.assertQuerysetEqual(
            response.context['latest_question_list'],
            ['<Question: Past question.>']
        )

    def test_index_view_with_two_past_questions(self):
        """
        The questions index page may display multiple questions.
        """
        create_question(question_text="Past question 1.", days=-30)
        create_question(question_text="Past question 2.", days=-5)
        response = self.client.get(reverse('polls:index'))
        self.assertQuerysetEqual(
            response.context['latest_question_list'],
            ['<Question: Past question 2.>', '<Question: Past question 1.>']
        )

Testing the DetailView

# polls/views.py
class DetailView(generic.DetailView):
    ...
    def get_queryset(self):
        """
        Excludes any questions that aren't published yet.
        """
        return Question.objects.filter(pub_date__lte=timezone.now())

# ./polls/tests.py
class QuestionIndexDetailTests(TestCase):
    def test_detail_view_with_a_future_question(self):
        """
        The detail view of a question with a pub_date in the future should
        return a 404 not found.
        """
        future_question = create_question(question_text='Future question.', days=5)
        url = reverse('polls:detail', args=(future_question.id,))
        response = self.client.get(url)
        self.assertEqual(response.status_code, 404)

    def test_detail_view_with_a_past_question(self):
        """
        The detail view of a question with a pub_date in the past should
        display the question's text.
        """
        past_question = create_question(question_text='Past Question.', days=-5)
        url = reverse('polls:detail', args=(past_question.id,))
        response = self.client.get(url)
        self.assertContains(response, past_question.question_text)

Ideas for more tests

우리는 ResultsView에 비슷한 get_queryset 메소드를 추가하고 해당 뷰에 대한 새로운 테스트 클래스를 작성해야 함. 그것은 우리가 방금 만든 것과 매우 유사. 사실 많은 반복이있을 것.

Test를 추가하면서 다른 방법으로 Apps을 개선 할 수도 있음.

예로 Choices 없는 사이트에 Questions 을 게시할 수 있다는 것은 어리석은 일. 따라서 우리의 견해는 이를 확인하고 그러한 Question 을 제외 할 수 있음. 우리의 Test는 선택없이 Question 을 작성한 다음 게시되지 않았으며 Choices 와 함께 비슷한 Question 을 작성하고 게시되었는지 테스트 함.

아마도 로그인 한 관리자는 일반 게시자가 아닌 게시되지 않은 질문을 볼 수 있어야 함. 다시 말해: 이 작업을 수행하기 위해 소프트웨어에 추가해야 하는 것은 무엇이든 먼저 테스트를 작성한 다음 코드가 테스트를 통과하게 하거나 테스트 코드를 먼저 작성한 다음 테스트를 작성하는 등의 테스트가 수반되어야 함.

특정 시점에서 테스트를 보고 코드가 테스트 팽창으로 고통 받고 있는지 궁금해 할 수 있음.

When testing, more is better

우리의 테스트가 통제를 벗어나는 것처럼 보일 수도 있음.

이 속도로 곧 우리의 응용 프로그램에서보다 우리의 테스트에서 더 많은 코드가 있을 것. 반복은 우리 코드의 우아한 간결함과 비교하여 미학적. 테스트 코드가 늘어나는 것에 대해서는 중요하지 않음. 그들이 자라게 하십시오.

대부분의 경우 테스트를 한 번 작성한 다음 잊어 버릴 수 있음. 계속해서 프로그램을 개발하면서 유용한 기능을 계속 수행 할 것. 때로는 테스트를 업데이트해야 함.

선택과 관련된 질문만 게시되도록 의견을 수정한다고 가정합니다. 이 경우 기존 테스트 중 상당수가 실패할 것.

즉, 최신 테스트를 수행하기 위해 어떤 테스트를 수정해야 하는지 정확하게 알려주므로 테스트가 스스로를 도와줌.

최악의 경우, 개발을 계속할 때 현재 중복되는 테스트가 있음을 알 수 있음.

그러나 그것은 문제가 아님.

중복 테스트는 좋은 일임.

검사가 현명하게 배열되어 있는한 관리가 어려워지지 않을 것.

좋은 규칙은 다음을 포함:

  • 각 model과 view의 별도의 TestClass

  • test하려는 각 조건의 집합에 대한 별도의 테스트 방법

  • 테스트 방법을 그 함수에 이름에 묘사

Future testing

현재는 테스트의 기본 사항에 대해서만 소개함.

Test Tools

  • 예로, 여기서 수행한 테스트에서는 model의 내부 논리와 view에서 정보를 게시하는 방법을 다루었지만 Selenium(tools)과 같은 "브라우저 내" framework를 사용하여 HTML이 실제로 browser에서 렌더링되는 방식을 test할 수 있음.

  • 이러한 tools를 사용하면 django 코드의 동작뿐만 아니라 예로 JavaScript를 확인가능.

  • Test가 Browser를 시작하고 인간이 운전하는 것처럼 사이트와 상호작용하기 시작하는 것이 매우 중요!!

  • Django에는 Selenium과 같은 도구와의 통합을 용이하게하기 위해 LiveServerTestCase 가 포함되어 있음.

복잡한 응용 프로그램

  • 지속적인 통합을 위해 모든 commit마다 자동으로

  • 테스트를 실행하여 품질 제어 자체가 적어도 부분적으로 자동화되도록 할 수 있음.

테스트되지 않은 apps 프로그램 부분을 탐지하는 좋은 방법

  • 코드 적용 범위를 확인하는 것.

이는 또한 깨지기 쉬운 코드나 심지어는 죽은 코드를 식별하는 데 도움이 됨.

코드를 test할 수 없다면 대개 코드를 Refactorying or Remove 해야 함.

Coverage는 죽은 코드를 확인하는 데 도움 됨.

Introducing automated testing

What are autumated tests?

한 번 테스트 세트를 작성한 다음 앱을 변경하면서 시간이 오래 걸리는 수동 테스트를 수행하지 않아도 원래 의도대로 코드가 작동하는지 확인할 수 있습니다.

Why you need to create tests

자동화 된 테스트를 만드는 데 어려움을 겪고 있으면 더 나은 결과를 얻지 못할 것.

Tests will save you time

시간 절약, 의존성 문제 테스트, 매번 또는 즉시 테스트 가능.

Tests don't just identify problems, they prevent them

테스트는 문제를 식별 뿐만 아니라 문제를 방지함.

Tests make your code more attractive

신뢰성 얻을 수 있음.

Tests help teams work together

테스트는 팀과의 협업을 돕는다.

장고 프로그래머로서 생계를 꾸려 나가려면 테스트를 잘해야 함!

Basic testing strategies

  • ""테스트 중심 개발"이라는 원칙을 따름.

  • 그들은 실제로 코드를 작성하기 전에 테스트를 작성.

  • 이것은 반 직관적으로 보일지 모르지만,

실제로는 대부분의 사람들이 자주 수행하는 것과 유사.

문제를 설명하고 문제를 해결하기 위해 코드를 작성합니다.

  • 수천 줄의 파이썬을 작성했다면, 테스트 할 대상을 선택하는 것이 쉽지 않을 수도 있음.

이 경우 다음 번에 변경 사항을 적용 할 때 새로운 기능을 추가하거나 버그를 수정할 때

첫 번째 테스트를 작성하는 것이 효과적.

Create a test to expose the bug

문제를 테스트하기 위해 shell에서 방금 수행 한 작업은 자동화 된 테스트에서 수행할 수 있는 작업이므로 자동화 된 테스트로 바꾸자. 애플리케이션의 테스트를 위한 일반적인 장소는 애플리케이션의 tests.py 파일에 있음. 테스트 시스템은 이름이 test로 시작하는 모든 파일에서 테스트를 자동으로 찾습니다. polls 응용 프로그램의 tests.py 파일에 다음을 입력하십시오.

# ./polls/test.py
import datetime

from django.utils import timezone
from django.test import TestCase

from .models import Question


class QuestionMethodTests(TestCase):

    def test_was_published_recently_with_future_question(self):
        """
        was_published_recently() should return False for questions whose
        pub_date is in the future.
        """
        time = timezone.now() + datetime.timedelta(days=30)
        future_question = Question(pub_date=time)
        self.assertIs(future_question.was_published_recently(), False)

여기에서 수행 한 작업은 pub_date 를 사용하여 Question 인스턴스를 생성하는 method로 django.test.TestCase 서브 클래스를 작성하는 것입니다. 그런 다음 was_published_recently() 의 결과를 확인합니다. 이 값은 False 여야합니다.

# ...
def was_published_recently(self):
    now = timezone.now()
    return now - datetime.timedelta(days=1) <= self.pub_date <= now

More comprehensive tests

하나의 method를 테스트하기 위해 Test method를 여러개 추가해야되는 당혹스러움.

# ./polls/tests.py

def test_was_published_recently_with_old_question(self):
    """
    was_published_recently() should return False for questions whose
    pub_date is older than 1 day.
    """
    time = timezone.now() - datetime.timedelta(days=30)
    old_question = Question(pub_date=time)
    self.assertIs(old_question.was_published_recently(), False)

def test_was_published_recently_with_recent_question(self):
    """
    was_published_recently() should return True for questions whose
    pub_date is within the last day.
    """
    time = timezone.now() - datetime.timedelta(hours=1)
    recent_question = Question(pub_date=time)
    self.assertIs(recent_question.was_published_recently(), True)

from django.urls import reverse

def create_question(question_text, days):
    """
    Creates a question with the given `question_text` and published the
    given number of `days` offset to now (negative for questions published
    in the past, positive for questions that have yet to be published).
    """
    time = timezone.now() + datetime.timedelta(days=days)
    return Question.objects.create(question_text=question_text, pub_date=time)


class QuestionViewTests(TestCase):
    def test_index_view_with_no_questions(self):
        """
        If no questions exist, an appropriate message should be displayed.
        """
        response = self.client.get(reverse('polls:index'))
        self.assertEqual(response.status_code, 200)
        self.assertContains(response, "No polls are available.")
        self.assertQuerysetEqual(response.context['latest_question_list'], [])

    def test_index_view_with_a_past_question(self):
        """
        Questions with a pub_date in the past should be displayed on the
        index page.
        """
        create_question(question_text="Past question.", days=-30)
        response = self.client.get(reverse('polls:index'))
        self.assertQuerysetEqual(
            response.context['latest_question_list'],
            ['<Question: Past question.>']
        )

    def test_index_view_with_a_future_question(self):
        """
        Questions with a pub_date in the future should not be displayed on
        the index page.
        """
        create_question(question_text="Future question.", days=30)
        response = self.client.get(reverse('polls:index'))
        self.assertContains(response, "No polls are available.")
        self.assertQuerysetEqual(response.context['latest_question_list'], [])

    def test_index_view_with_future_question_and_past_question(self):
        """
        Even if both past and future questions exist, only past questions
        should be displayed.
        """
        create_question(question_text="Past question.", days=-30)
        create_question(question_text="Future question.", days=30)
        response = self.client.get(reverse('polls:index'))
        self.assertQuerysetEqual(
            response.context['latest_question_list'],
            ['<Question: Past question.>']
        )

    def test_index_view_with_two_past_questions(self):
        """
        The questions index page may display multiple questions.
        """
        create_question(question_text="Past question 1.", days=-30)
        create_question(question_text="Past question 2.", days=-5)
        response = self.client.get(reverse('polls:index'))
        self.assertQuerysetEqual(
            response.context['latest_question_list'],
            ['<Question: Past question 2.>', '<Question: Past question 1.>']
        )

Ideas for more tests

우리는 ResultsView에 비슷한 get_queryset 메소드를 추가하고 해당 뷰에 대한 새로운 테스트 클래스를 작성해야 함. 그것은 우리가 방금 만든 것과 매우 유사. 사실 많은 반복이있을 것.

Test를 추가하면서 다른 방법으로 Apps을 개선 할 수도 있음.

예로 Choices 없는 사이트에 Questions 을 게시할 수 있다는 것은 어리석은 일. 따라서 우리의 견해는 이를 확인하고 그러한 Question 을 제외 할 수 있음. 우리의 Test는 선택없이 Question 을 작성한 다음 게시되지 않았으며 Choices 와 함께 비슷한 Question 을 작성하고 게시되었는지 테스트 함.

아마도 로그인 한 관리자는 일반 게시자가 아닌 게시되지 않은 질문을 볼 수 있어야 함. 다시 말해: 이 작업을 수행하기 위해 소프트웨어에 추가해야 하는 것은 무엇이든 먼저 테스트를 작성한 다음 코드가 테스트를 통과하게 하거나 테스트 코드를 먼저 작성한 다음 테스트를 작성하는 등의 테스트가 수반되어야 함.

특정 시점에서 테스트를 보고 코드가 테스트 팽창으로 고통 받고 있는지 궁금해 할 수 있음.

When testing, more is better

우리의 테스트가 통제를 벗어나는 것처럼 보일 수도 있음.

이 속도로 곧 우리의 응용 프로그램에서보다 우리의 테스트에서 더 많은 코드가 있을 것. 반복은 우리 코드의 우아한 간결함과 비교하여 미학적. 테스트 코드가 늘어나는 것에 대해서는 중요하지 않음. 그들이 자라게 하십시오.

대부분의 경우 테스트를 한 번 작성한 다음 잊어 버릴 수 있음. 계속해서 프로그램을 개발하면서 유용한 기능을 계속 수행 할 것. 때로는 테스트를 업데이트해야 함.

선택과 관련된 질문만 게시되도록 의견을 수정한다고 가정합니다. 이 경우 기존 테스트 중 상당수가 실패할 것.

즉, 최신 테스트를 수행하기 위해 어떤 테스트를 수정해야 하는지 정확하게 알려주므로 테스트가 스스로를 도와줌.

최악의 경우, 개발을 계속할 때 현재 중복되는 테스트가 있음을 알 수 있음.

그러나 그것은 문제가 아님.

중복 테스트는 좋은 일임.

검사가 현명하게 배열되어 있는한 관리가 어려워지지 않을 것.

좋은 규칙은 다음을 포함:

  • 각 model과 view의 별도의 TestClass

  • test하려는 각 조건의 집합에 대한 별도의 테스트 방법

  • 테스트 방법을 그 함수에 이름에 묘사

Future testing

현재는 테스트의 기본 사항에 대해서만 소개함.

Test Tools

  • 예로, 여기서 수행한 테스트에서는 model의 내부 논리와 view에서 정보를 게시하는 방법을 다루었지만 Selenium(tools)과 같은 "브라우저 내" framework를 사용하여 HTML이 실제로 browser에서 렌더링되는 방식을 test할 수 있음.

  • 이러한 tools를 사용하면 django 코드의 동작뿐만 아니라 예로 JavaScript를 확인가능.

  • Test가 Browser를 시작하고 인간이 운전하는 것처럼 사이트와 상호작용하기 시작하는 것이 매우 중요!!

  • Django에는 Selenium과 같은 도구와의 통합을 용이하게하기 위해 LiveServerTestCase 가 포함되어 있음.

복잡한 응용 프로그램

  • 지속적인 통합을 위해 모든 commit마다 자동으로

  • 테스트를 실행하여 품질 제어 자체가 적어도 부분적으로 자동화되도록 할 수 있음.

테스트되지 않은 apps 프로그램 부분을 탐지하는 좋은 방법

  • 코드 적용 범위를 확인하는 것.

이는 또한 깨지기 쉬운 코드나 심지어는 죽은 코드를 식별하는 데 도움이 됨.

코드를 test할 수 없다면 대개 코드를 Refactorying or Remove 해야 함.

Coverage는 죽은 코드를 확인하는 데 도움 됨.

  • 얼마나 많은 소스 코드가 테스트되었는지를 설명.

  • 코드의 어떤 부분이 테스트에 의해 실행되고 어떤 부분이 테스트에 의해 실행되는지를 보여줌.

  • 테스트 애플리케이션의 중요한 부분이므로 테스트 범위를 확인하는 것이 좋음.
