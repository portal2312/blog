Customize the admin form

Django는 admin.site.register(Question) 로 Question model을 등록함으로써 기본 form 표현을 구성 할 수 있음. admin 양식이 어떻게 보이고 작동하는지 사용자 정의하려는 경우가 많음. 객체를 등록 할 때 Django에 원하는 옵션을 알려줌으로써 이를 수행 할 것입니다. 편집 form의 fields를 재정렬하여 이것이 어떻게 작동하는지 보겠음. admin.site.register(Question) 행을 다음으로 대체하십시오.

# ./polls/admin.py
from django.contrib import admin
from .models import Question


class QuestionAdmin(admin.ModelAdmin):
    fields = ['pub_date', 'question_text']

admin.site.register(Question, QuestionAdmin)

이 pattern을 따라 가면 model admin class를 만든 다음 model의 admin options을 변경해야 할 때마다 admin.site.register() 의 대한 두 번째 인수로 전달.

위의 특별한 변경 사항은 "Publication date"이 "Question" field 앞에 오게 만듬:

  • 이미지 생략

이것은 두 필드만으로는 인상적이지 않지만 수십 개의 fields가 있는 admin forms의 경우 직관적인 order을 선택하는 것이 사용 편리 성의 중요한 부분.

수십 개의 fields가 있는 forms에 관해서는 forms을 fieldsets으로 분할하는 것이 좋음:

# ./polls/admin.py
from django.contrib import admin
from .models import Question


class QuestionAdmin(admin.ModelAdmin):
    fieldsets = [
        (None,               {'fields': ['question_text']}),
        ('Date information', {'fields': ['pub_date']}),
    ]

admin.site.register(Question, QuestionAdmin)

fieldsets 의 각 tuple의 첫 번째 element는 필드 세트의 title임.

Adding related objects

OK, Question Admin 페이지가 있지만, Question 에 여러가지 Choices 사항이 있으며, admin page에 선택사항이 표시되지 않음.

아직.

이 문제를 해결하는 데는 두 가지 방법이 있습니다. 첫 번째는 Question 과 마찬가지로 관리자에게 Choice 를 등록하는 것:

# polls/admin.py

from django.contrib import admin
from .models import Choice, Question
# ...
admin.site.register(Choice)

이제 "Choices"은 django admin가 사용할 수 있는 옵션. 'Add choice' 양식은 다음과 같음.

  • 이미지 생략

이 양식엔, "Question" 필드는 Database의 모든 질문을 포함하는 <select> 상자임. Django는 ForeignKey가 admin에서 <select> box로 표시되어야 함을 암. 우리의 경우, 단 하나의 질문 만이 시점에 존재.

또한 "Question"옆의 "Add Another" link에 주목. 다른 개체에 대한 ForeignKey 관계가 있는 모든 개체는 무료로 이 개체를 가져옴. 'Add Another'를 클릭하면 'Add Question' 양식이 있는 팝업 창이 나타남.

해당 창에 Question 추가후 "저장" 클릭시 django는 Question 을 DB에 저장하고 보고 있는 "선택 항목 추가" 양식에서 선택된 선택 항목으로 동적으로 추가함. 그러나 실제로 이것은 Choice 객체를 시스템에 추가하는 비효율적인 방법임.

Question 객체를 생성 할 때 Choices를 직접 추가 할 수 있다면 더 좋을 것. 만들어 보기.

Choice models에 대한 register() 호출을 제거.

그런 다음 Question 등록 코드를 편집하여 다음을 읽으십시오:

# ./polls/admin.py
from django.contrib import admin
from .models import Choice, Question


class ChoiceInline(admin.StackedInline):
    model = Choice
    extra = 3


class QuestionAdmin(admin.ModelAdmin):
    fieldsets = [
        (None, {'fields': ['question_text']}),
        ('Date information', {'fields': ['pub_date'], 'classes': ['collapse']}),
    ]
    inlines = [ChoiceInline]


admin.site.register(Question, QuestionAdmin)

이것은 Django에게 알려줌: "Choice 항목은 Question 관리자 페이지에서 편집됨.

기본적으로 3가지 선택 항목에 대해 충분한 필드를 제공하십시오."

모양을 보려면 "Add question" page를 load.

  • 이미지 생략

그것은 다음과 같이 작동: 관련 선택 사항을 위한 세 개의 슬롯이 있음 - extra 에 의해 열거 됨 - 이미 생성 된 객체의 "Change" page로 돌아올 때마다 또 다른 세 개의 슬롯이 추가로 생김.

3개의 현재 슬롯 끝에 "Add another Choice" link가 있음. 그것을 클릭하면 새로운 슬롯이 추가됨. 추가 된 슬롯을 제거하려면 추가 된 슬롯의 오른쪽 상단에있는 X를 click. 원래의 세 슬롯은 제거 할 수 없습니다. 이 이미지는 추가 된 슬롯을 보여줍니다.

  • 이미지 생략

class ChoiceInline(admin.TabularInline):

Admin List

  • default=Django는 각 객체의 str()을 표시.

  • list_display admin options (개별 필드 표시)

    • Field name을 Tuple()로 정의해서 Page의 Columns으로 표시.

was_published_recently header의 경우를 제외하고는 임의의 메소드 출력에 의한 정렬이 지원되지 않으므로 열 머리글을 클릭하여 해당 값을 기준으로 정렬 가능. 또한 was_published_recently 의 열 머리글은 기본적으로 메서드 이름(밑줄이 공백으로 바뀜)이며 각 줄에는 출력의 문자열 표현이 포함됨.

# ./polls/models.py
class Question(models.Model):
    # ...
    def was_published_recently(self):
        now = timezone.now()
        return now - datetime.timedelta(days=1) <= self.pub_date <= now
    was_published_recently.admin_order_field = 'pub_date'
    was_published_recently.boolean = True
    was_published_recently.short_description = 'Published recently?'

이러한 메서드 속성에 대한 자세한 내용은 list_display 참조.

Customize the admin look and feel

분명히 각 관리자 페이지 상단에 "Django administration"(그냥 텍스트 표시) Django의 템플릿 시스템을 사용하면 쉽게 변경 가능. Django admin는 Django 자체에 의해 구동되며, 인터페이스는 Django 자신의 template 시스템을 사용함.

Customizing your projects\'s templates

Project Dir(manage.py 있는 Dir)에 templates Dir를 만듬. Templates은 Django가 접근 가능한 파일 시스템 어디서나 사용할 수 있음 (Django는 서버를 실행하는 모든 사용자로 실행됨). 그러나 templates 유지하는 것은 projects 내에서 따라야하는 좋은 규칙임.

설정 파일(mysite/settings.py) 열고 TEMPLATES 설정에 DIRS 에 옵션 추가.

# mysite/settings.py

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [os.path.join(BASE_DIR, 'templates')],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
            ],
        },
    },
]

django 자체의 소스 코드(django/contrib/admin/templates)에 있는 기본 django 관리 templates Dir 내의 admin/base_site.html 을 templates 안에 admin Dir에 복사.

* Where are the Django source files?

$ python -c "import django; print(django.__path__)"
# /usr/local/lib/python3.5/site-packages

$ cp -rvp (생략)/django/contrib/admin/templates/admin/base_site.html \
          $HOME/mysite/templates/admin/

그런 다음 파일을 편집하고 {{ site_header | default : _('Django administration') }} (중괄호 포함)를 자신의 사이트 이름으로 바꿉니다. 다음과 같은 코드 섹션을 작성해야 함.

<!--./mysite/templates/admin/base_site.html-->

<!--(...)-->
{% block branding %}
<h1 id="site-name">
    <a href="{% url 'admin:index' %}">
        <!--{{ site_header|default:_('Django administration') }}-->
        Polls Administration
    </a>
</h1>
{% endblock %}
<!--(...)-->

이 방법을 사용하여 템플릿을 재정의하는 방법을 학습합니다. 실제 프로젝트에서는 django.contrib.admin.AdminSite.site_header 속성을 사용하여 이 특정 사용자 정의를보다 쉽게 ​​만들 수 있음.

이 template 파일에는 {\% block branding \%} 및 {{ title }} 과 같은 text가 많이 포함되어 있음. {\% 와 {{ tags는 Django의 template 언어의 일부 임. Django가 admin/base_site.html 을 렌더링할 때, 이 template 언어는 Tutorial 3 에서 보았듯이 최종 HTML 페이지를 생성하도록 평가됨.

Django의 기본 admin 템플릿은 무시할 수 있습니다. 템플릿을 덮어 쓰려면 base_site.html 과 동일한 작업을 수행하십시오 - 기본 Dir에서 사용자 정의 Dir로 복사 한 다음 변경하십시오.

Customizing your application\'s templates

DIRS 이 기본적으로 비어 있다면 Django 기본 관리 templates을 어떻게 찾나? 대답은 APP_DIRS=True 설정되어 있기 때문. Django는 각 application package 내의 templates/ 하위 디렉토리를 자동으로 찾아서 fallback으로 사용(잊지마라, django.contrib.admin 은 application).

polls application은 매우 복잡하지 않으며 사용자 정의 관리 템플릿이 필요하지 않음. 그러나 Django의 표준 관리 템플릿을 좀 더 정교하고 필수적으로 수정하여 기능을 일부 추가시 templates 은 projects 보다 application 쪽을 수정하는 것이 더 현명. 그렇게 하면 새 프로젝트에 polls application을 포함시킬 수 있고 필요한 맞춤 템플릿을 찾을 수 있음.

Customize the admin index page

비슷한 내용으로 Django admin index page의 모양과 느낌을 사용자 정의하고 싶을 수도 있음.

기본적으로, admin application에 등록된 INSTALLED_APPS 의 모든 apps을 알파벳순으로 표시. 레이아웃을 크게 변경하고자 할 수 있음. 어쨌든 index는 아마 admin의 가장 중요한 page이고, 사용하기 쉬워야 함.

사용자 정의할 templates는 admin/index.html 임. (이전 섹션의 admin/base_site.html 과 동일하게 하십시오 - 기본 디렉토리에서 사용자 정의 templates Dir로 복사). 파일을 편집하면 app_list 라는 templates 변수가 사용 된 것을 볼 수 있음. 이 변수는 설치된 모든 Django App을 포함함. 이를 사용하는 대신 최선의 방법이라고 생각하는 방식으로 개체 별 관리 페이지에 대한 링크를 hard-code 할 수 있음.

# ./polls/admin.py
from django.contrib import admin
from .models import Choice, Question


class ChoiceInline(admin.StackedInline):
    model = Choice
    extra = 3


class QuestionAdmin(admin.ModelAdmin):
    fieldsets = [
        (None, {'fields': ['question_text']}),
        ('Date information', {'fields': ['pub_date'], 'classes': ['collapse']}),
    ]
    inlines = [ChoiceInline]


admin.site.register(Question, QuestionAdmin)

이것은 Django에게 알려줌: "Choice 항목은 Question 관리자 페이지에서 편집됨.

기본적으로 3가지 선택 항목에 대해 충분한 필드를 제공하십시오."

모양을 보려면 "Add question" page를 load.

그것은 다음과 같이 작동: 관련 선택 사항을 위한 세 개의 슬롯이 있음 - extra 에 의해 열거 됨 - 이미 생성 된 객체의 "Change" page로 돌아올 때마다 또 다른 세 개의 슬롯이 추가로 생김.

3개의 현재 슬롯 끝에 "Add another Choice" link가 있음. 그것을 클릭하면 새로운 슬롯이 추가됨. 추가 된 슬롯을 제거하려면 추가 된 슬롯의 오른쪽 상단에있는 X를 click. 원래의 세 슬롯은 제거 할 수 없습니다. 이 이미지는 추가 된 슬롯을 보여줍니다.

class ChoiceInline(admin.TabularInline):

    • Field name을 Tuple()로 정의해서 Page의 Columns으로 표시.

was_published_recently header의 경우를 제외하고는 임의의 메소드 출력에 의한 정렬이 지원되지 않으므로 열 머리글을 클릭하여 해당 값을 기준으로 정렬 가능. 또한 was_published_recently 의 열 머리글은 기본적으로 메서드 이름(밑줄이 공백으로 바뀜)이며 각 줄에는 출력의 문자열 표현이 포함됨.

# ./polls/models.py
class Question(models.Model):
    # ...
    def was_published_recently(self):
        now = timezone.now()
        return now - datetime.timedelta(days=1) <= self.pub_date <= now
    was_published_recently.admin_order_field = 'pub_date'
    was_published_recently.boolean = True
    was_published_recently.short_description = 'Published recently?'

이러한 메서드 속성에 대한 자세한 내용은 list_display 참조.

Customizing your projects\'s templates

Project Dir(manage.py 있는 Dir)에 templates Dir를 만듬. Templates은 Django가 접근 가능한 파일 시스템 어디서나 사용할 수 있음 (Django는 서버를 실행하는 모든 사용자로 실행됨). 그러나 templates 유지하는 것은 projects 내에서 따라야하는 좋은 규칙임.

설정 파일(mysite/settings.py) 열고 TEMPLATES 설정에 DIRS 에 옵션 추가.

# mysite/settings.py

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [os.path.join(BASE_DIR, 'templates')],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
            ],
        },
    },
]

django 자체의 소스 코드(django/contrib/admin/templates)에 있는 기본 django 관리 templates Dir 내의 admin/base_site.html 을 templates 안에 admin Dir에 복사.

* Where are the Django source files?

$ python -c "import django; print(django.__path__)"
# /usr/local/lib/python3.5/site-packages

$ cp -rvp (생략)/django/contrib/admin/templates/admin/base_site.html \
          $HOME/mysite/templates/admin/

그런 다음 파일을 편집하고 {{ site_header | default : _('Django administration') }} (중괄호 포함)를 자신의 사이트 이름으로 바꿉니다. 다음과 같은 코드 섹션을 작성해야 함.

<!--./mysite/templates/admin/base_site.html-->

<!--(...)-->
{% block branding %}
<h1 id="site-name">
    <a href="{% url 'admin:index' %}">
        <!--{{ site_header|default:_('Django administration') }}-->
        Polls Administration
    </a>
</h1>
{% endblock %}
<!--(...)-->

이 방법을 사용하여 템플릿을 재정의하는 방법을 학습합니다. 실제 프로젝트에서는 django.contrib.admin.AdminSite.site_header 속성을 사용하여 이 특정 사용자 정의를보다 쉽게 ​​만들 수 있음.

이 template 파일에는 {\% block branding \%} 및 {{ title }} 과 같은 text가 많이 포함되어 있음. {\% 와 {{ tags는 Django의 template 언어의 일부 임. Django가 admin/base_site.html 을 렌더링할 때, 이 template 언어는 Tutorial 3 에서 보았듯이 최종 HTML 페이지를 생성하도록 평가됨.

Django의 기본 admin 템플릿은 무시할 수 있습니다. 템플릿을 덮어 쓰려면 base_site.html 과 동일한 작업을 수행하십시오 - 기본 Dir에서 사용자 정의 Dir로 복사 한 다음 변경하십시오.

Customizing your application\'s templates

DIRS 이 기본적으로 비어 있다면 Django 기본 관리 templates을 어떻게 찾나? 대답은 APP_DIRS=True 설정되어 있기 때문. Django는 각 application package 내의 templates/ 하위 디렉토리를 자동으로 찾아서 fallback으로 사용(잊지마라, django.contrib.admin 은 application).

polls application은 매우 복잡하지 않으며 사용자 정의 관리 템플릿이 필요하지 않음. 그러나 Django의 표준 관리 템플릿을 좀 더 정교하고 필수적으로 수정하여 기능을 일부 추가시 templates 은 projects 보다 application 쪽을 수정하는 것이 더 현명. 그렇게 하면 새 프로젝트에 polls application을 포함시킬 수 있고 필요한 맞춤 템플릿을 찾을 수 있음.

Customize the admin index page

비슷한 내용으로 Django admin index page의 모양과 느낌을 사용자 정의하고 싶을 수도 있음.

기본적으로, admin application에 등록된 INSTALLED_APPS 의 모든 apps을 알파벳순으로 표시. 레이아웃을 크게 변경하고자 할 수 있음. 어쨌든 index는 아마 admin의 가장 중요한 page이고, 사용하기 쉬워야 함.

사용자 정의할 templates는 admin/index.html 임. (이전 섹션의 admin/base_site.html 과 동일하게 하십시오 - 기본 디렉토리에서 사용자 정의 templates Dir로 복사). 파일을 편집하면 app_list 라는 templates 변수가 사용 된 것을 볼 수 있음. 이 변수는 설치된 모든 Django App을 포함함. 이를 사용하는 대신 최선의 방법이라고 생각하는 방식으로 개체 별 관리 페이지에 대한 링크를 hard-code 할 수 있음.
