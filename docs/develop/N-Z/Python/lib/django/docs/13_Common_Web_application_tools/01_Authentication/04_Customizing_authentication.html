Django와 함께 제공되는 인증은 대부분의 일반적인 경우에 충분하지만 기본 설정으로 충족되지 않아도 됩니다. 프로젝트에서 인증을 사용자 정의하려면 제공된 시스템의 어떤 지점을 확장 또는 교체 할 수 있는지 이해해야 합니다. 이 문서는 인증 시스템을 사용자 정의하는 방법에 대한 세부 사항을 제공합니다.

Authentication backends는 사용자 모델과 함께 저장된 사용자 이름과 비밀번호가 장고의 기본값과 다른 서비스에 대해 인증되어야 할 때 확장 가능한 시스템을 제공합니다.

Django의 인증 시스템을 통해 확인할 수 있는 사용자 정의 권한을 모델에 부여 할 수 있습니다.

기본 사용자 모델을 확장하거나 완전히 사용자 지정된 모델을 대체 할 수 있습니다.

Other authentication sources

다른 인증 소스 (사용자 이름과 비밀번호 또는 인증 메소드의 또 다른 소스)에 연결해야 할 필요가있을 수 있습니다.

예를 들어, 회사에는 이미 모든 직원에 대한 사용자 이름과 암호를 저장하는 LDAP 설정이있을 수 있습니다. 사용자가 LDAP 및 Django 기반 응용 프로그램에서 별도의 계정을 가지고 있다면 네트워크 관리자와 사용자 모두에게 번거로운 작업이 될 수 있습니다.

따라서 이런 상황을 처리하기 위해 Django 인증 시스템을 사용하면 다른 인증 소스를 연결할 수 있습니다. Django의 기본 데이터베이스 기반 구성표를 무시하거나 다른 시스템과 함께 기본 시스템을 사용할 수 있습니다.

Django에 포함 된 인증 백엔드에 대한 정보는 authentication backend reference를 참조하십시오.

Specifying authentication backends

인증 백앤드 지정하기

배후에서 장고는 인증을 위해 검사하는 "authentication backends" 목록을 유지합니다. 누군가가 django.contrib.auth.authenticate() 를 호출 할 때 - 사용자 로그인 방법에 설명 된대로 - django는 모든 인증 백엔드에서 인증을 시도합니다. 첫 번째 인증 방법이 실패하면 Django는 모든 백엔드가 시도 될 때까지 두 번째 인증 방법을 시도합니다. 사용할 인증 백엔드 목록은 AUTHENTICATION_BACKENDS 설정에 지정됩니다. 이것은 인증 방법을 알고있는 Python 클래스를 가리키는 Python 경로 이름 목록이어야합니다. 이 클래스는 파이썬 경로의 어느 곳에 나있을 수 있습니다. 기본적으로 AUTHENTICATION_BACKENDS 는 다음과 같이 설정됩니다.

['django.contrib.auth.backends.ModelBackend']

이것은 Django 사용자 데이터베이스를 검사하고 내장 된 권한을 쿼리하는 기본 인증 백엔드입니다. 속도 제한 메커니즘을 통한 무차별 공격에 대한 보호 기능을 제공하지 않습니다. 사용자 지정 인증 백엔드에서 자체 속도 제한 메커니즘을 구현하거나 대부분의 웹 서버에서 제공하는 메커니즘을 사용할 수 있습니다.

AUTHENTICATION_BACKENDS의 순서는 중요합니다. 따라서 여러 개의 백엔드에서 동일한 사용자 이름과 비밀번호가 유효한 경우 Django는 첫 번째 긍정적 인 일치에서 처리를 중단합니다.

백엔드에서 PermissionDenied 예외가 발생하면 인증이 즉시 실패합니다. Django는 뒤 따르는 백엔드를 확인하지 않습니다.

Note

사용자가 인증을 하면 Django는 백엔드를 사용하여 사용자의 세션에서 사용자를 인증하고 현재 인증 된 사용자에 대한 액세스가 필요할 때마다 해당 백엔드를 해당 세션 기간 동안 다시 사용합니다. 이는 인증 소스가 세션별로 캐시된다는 것을 의미하므로 AUTHENTICATION_BACKENDS를 변경하면 사용자가 다른 방법으로 재인증하도록해야하는 경우 세션 데이터를 지워야합니다. 간단한 방법은 단순히 Session.objects.all().delete() 를 실행하는 것입니다.

Extending the existing User model

사용자 모델을 대체하지 않고 기본 User model을 확장하는 데는 두 가지 방법이 있습니다.

필요한 변경 사항이 순수한 동작이며 데이터베이스에 저장된 내용을 변경할 필요가 없는 경우 User를 기반으로 proxy model을 만들 수 있습니다. 따라서 기본 주문, 사용자 지정 관리자 또는 사용자 지정 모델 방법을 비롯한 프록시 모델에서 제공하는 기능을 사용할 수 있습니다.

User와 관련된 정보를 저장하려는 경우 OneToOneField를 사용하여 추가 정보 필드가 포함 된 모델을 만들 수 있습니다. 이 일대일 모델은 종종 사이트 사용자에 대한 비인증 관련 정보를 저장할 수 있으므로 Profile model이라고도 합니다.

예를 들어 Employee 모델을 만들 수 있습니다:

from django.contrib.auth.models import User

class Employee(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    department = models.CharField(max_length=100)

기존 Employee Fred Smith가 User와 Employee 모델을 모두 가지고 있다고 가정하면 Django의 표준 관련 모델 규칙을 사용하여 관련 정보에 액세스 할 수 있습니다:

u = User.objects.get(username='fsmith')
freds_department = u.employee.department

프로필의 필드를 관리자의 사용자 페이지에 추가하려면 InlineModelAdmin을 앱의 admin.py에서 정의하고 이를 User 클래스에 등록된 [UserAdmin] 클래스에 추가하십시오:

from django.contrib import admin
from django.contrib.auth.admin import UserAdmin as BaseUserAdmin
from django.contrib.auth.models import User

from my_user_profile_app.models import Employee

# Define an inline admin descriptor for Employee model
# which acts a bit like a singleton
class EmployeeInline(admin.StackedInline):
    model = Employee
    can_delete = False
    verbose_name_plural = 'employee'

# Define a new User admin
class UserAdmin(BaseUserAdmin):
    inlines = (EmployeeInline,)

# Re-register UserAdmin
admin.site.unregister(User)
admin.site.register(User, UserAdmin)

이 프로파일 모델은 어떤 방식으로도 특별하지 않습니다. 단지 사용자 모델과 일대일로 연결되는 Django 모델 일뿐입니다. 따라서 사용자 생성시 자동 생성되지 않지만 [django.db.models.signals.post_save]를 사용하여 관련 모델을 적절히 작성하거나 업데이트 할 수 있습니다.

관련 모델을 사용하면 추가 쿼리 또는 조인을 통해 관련 데이터를 검색 할 수 있습니다. 필요에 따라 관련 필드가 포함 된 사용자 정의 사용자 모델이 더 나은 옵션 일 수 있지만 프로젝트 앱의 기본 사용자 모델과의 기존 관계로 인해 추가 데이터베이스로드가 정당화 될 수 있습니다.

관련 모델을 사용하면 추가 쿼리 또는 조인을 통해 관련 데이터를 검색 할 수 있습니다. 필요에 따라 관련 필드가 포함 된 사용자 정의 사용자 모델이 더 나은 옵션 일 수 있지만 프로젝트 앱의 기본 사용자 모델과의 기존 관계로 인해 추가 데이터베이스로드가 정당화 될 수 있습니다.
