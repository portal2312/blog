Branch 란?

  • Commit 완료시

    • Commit Objects
    • 메타데이터와 'Root Tree'를 가르키는 Pointer.
    • 현 Staging Area에 있는 데이터의 스냅샷에 대한 포인터, author 나 커밋 메시지 같은 메타데이터, 이전 커밋에 대한 포인터 등을 포함하는 커밋 개체(커밋 Object)를 저장한다.
    • 이전 커밋 포인터가 있어서 현재 커밋이 무엇을 기준으로 바뀌었는지를 알 수 있다.
    • 최초 커밋을 제외한 나머지 커밋은 이전 커밋 포인터가 적어도 하나씩 있고 브랜치를 합친 Merge 커밋 같은 경우에는 이전 커밋 포인터가 여러 개 있다.
    • Tree Objects : Files, Directory structure.
    • 각 Files Blob 객체들
    • Blob : 파일을 Stage 하면 Git 저장소에 파일을 저장.
    • File 마다 생성됨, CheckSum, SHA-1 로 되어있음.
    • 최초 commit 시 master 이름의 Branch 생성됨.

    'master' Branch 는 특별하지 않다.

    git init 명령으로 초기화 때 자동으로 만들어진게 이유가 끝이다. - commit 을 만들 때 마다 Branch가 자동으로 가장 마지막 commit 을 가르킴.

new Branch 생성하기

git branch <branch-name>

  • branch 는 HEAD 라는 pointer 가 있음. HEAD 는 현재 branch 위치를 가르킴.

  • git branch 시 수동으로 HEAD 를 잡아줘야 함.

  • 확인하기

  git log --oneline --graph --all --decorate  # --decorate 를 통해 볼 수 있음

Branch 이동하기

git checkout <branch-name>

  • branch 이동하면 Working Dir의 Files 변경됨 :

    • 브랜치에서 가장 마지막으로 했던 작업 내용으로 변경.
    • 파일 변경시 문제시 브랜치를 이동 불가능한 경우 브랜치 이동 명령을 수행하지 않음.
    • Tracked(unmodified, modified, staged) 만 변경.

  • 브랜치를 만들어야 하면 프로젝트를 통째로 복사해야 하는 다른 버전 관리 도구와 Git의 차이는 극명하다. 통째로 복사하는 작업은 프로젝트 크기에 따라 다르겠지만 수십 초에서 수십 분까지 걸린다. 그에 비해 Git은 순식간이다. 게다가 커밋을 할 때마다 이전 커밋의 정보를 저장하기 때문에 Merge 할 때 어디서부터(Merge Base) 합쳐야 하는지 안다.

  • Branch 와 Merge 의 보통의 경우

  1. 작업중

  2. new issue 처리

  3. branch

  • Hotfix

  2. create Hotfix branch

  3. Fixed Hotfix branch > Move Production > Merge Production

  4. 다시 작업중인 brach 로 가서 일 진행한다.

Branch의 기초

  • create new Branch, checkout branch 를 동시에 하기 bash git checkout -b <new-branch-name>

  • branch 삭제 bash $ git branch -d <branch-name>

Merge 의 기초

$ git checkout -b hotfix    # hotfix branch 를 만들면서 동시  checkout 함
$ vi hotfix.md
$ git commit -a -v -m 'description'

$ git checkout master    # checkout 으로 master 로 이동해서 기준을 선택하고
$ git merge hotfix       # master <- hotfix를 merge 함

Updating 2f54cf3..7531d07
Fast-forward    # fast-forward Merge 방식
 hotfix.txt | 1 +
 1 file changed, 1 insertion(+)
 create mode 100644 hotfix.txt


$ git branch -d hotfix    # 'hotfix' branch delete

  • Merge

    • 3-way
    • 현재 Branch가 가리키는 Commit 이 Merge 할 Branch의 조상이 아닌 경우.
    • 각 Branch가 가리키는 Commit 두 개와 공통 조상 하나를 사용.
    • fast-forward
    • Merge 할 브랜치가 가리키는 커밋이 현 브랜치 커밋의 Upstream 브랜치인 경우.
    • 브랜치 포인터는 최신 커밋으로 이동한다.
    • A 브랜치에서 다른 B 브랜치를 Merge 할 때 B가 A 이후의 커밋을 가리키고 있으면 그저 A가 B의 커밋을 가리키게 할 뿐이다.

  • 큰 Projects 에서도 자주 사용됨.

  • proposed or pu (proposed updates) : merge 단계가 아닌 것을 우선 merge.

Topic Branch

  • 프로젝트 크기에 상관없음.

  • 짧은 호흡의 branch.

  • iss1, hotfix 등

  • branch checkout 후 commit을 해야 마지막 저장이라고 생각하면 됨.

Remote Branch 

git remote show <remote-name>  # remote Refs 조회

git remote -v  # 간단하게 보기

  • remote Refs : remote repository 에 있는 pointer reference. (branch, tag 등등)

  • 보통 Remote Tracking Branch 를 사용.

  • Remote Tracking Branch

    • 리모트 브랜치를 추적하는 브랜치.
    • Local 에 있지만 움직일 수 없다.
    • remote Server 연결할 때마다 리모트 브랜치에 따라서 자동으로 움직일 뿐. (북마크)
    • 리모트 저장소에 마지막으로 연결했던 순간에 브랜치가 무슨 커밋을 가리키고 있었는지를 나타낸다.
    • 리모트 브랜치의 이름은 (remote)/(branch) 형식.
    • 리모트 저장소 origin의 master 브랜치를 보고 싶다면 origin/master 브랜치 확인.

  • 'origin' 의미 없음, 변경가능하나 remote repository branch 위치 파악하기 위해 안하는게 좋을 듯. bash git clone -o booyah -> booyah/master

  • Remote Repository 추가하기, fetch로 저장해서 git으로 관리.

git remote add <remote-name> <URL>
git fetch <remote-name>


$ git remote add office http://localhost/mks/office.git
# office가 origin/master 로 부터 나왔다면 git log 를 통해 연결된게 확인 된다.
$ git fetch office
$ git log --all --decorate --graph --oneline

Branch 추적

  • Tracking Branch (= upstream Branch)

    • Remote Tracking Branch 를 checkout 으로 만든 Branch.
    • git push/pull 하면 자동으로 함.
git pull  # 자동 merge

# 생성
git log --oneline --decorate                      # 확인 존재유무
git checkout --track <remote-tracking-branch>     # 생성
# 생성 <remote-tracking-branch> -> <new-branch-name>
git checkout -b <new-branch-name> <remote-tracking-branch>

# 추적
git checkout <target-branch-name>       # 이동 <target-branch-name>
git branch -u <remote-tracking-branch>  # 생성 <remote-tracking-branch>
git branch --set-upstream-to=<remote-tracking-branch>  # 위와 동일

# 확인
git branch -vv

# 삭제 (사용하지 않을 경우)
git branch -d <remote-tracking-branch>


$ git log --oneline --decorate
$ git checkout --track origin/iss001
$ git checkout iss001
$ git branch --set-upstream-to=origin/iss001
$ git branch -vv

Rebase

  • 한 branch에서 다른 branch로 합치는 방법. (Merge, Rebase)

  • 차이는 Rebase 는 Branch 를 만들지 않음.

  • master branch 로 merge 하기 전 작업 중인 다른 branch 들을 정리 목적으로 사용.

Rebase Basic

git checkout <topic-branch-name>
git rebase <base-branch-name>
git checkout <base-branch-name>
git merge <topic-branch-name>
git log --decorate --oneline  # 이 때 iss1 이 tag 존재시 iss1이 남아 있음.


$ git checkout iss1
$ git rebase master
$ git checkout master
$ git merge iss1
$ git log --decorate --oneline  # 이 때 iss1 이 tag 존재시 iss1이 남아 있음.

# Tag 존재시
$ git log --decorate --oneline
# * 073d13c (HEAD, tag: c4", master, iss1) c4
# * a2b1670 (tag: c3) c3
# | * 297b953 (tag: c4) c4
# |/
# * e1e72c5 (tag: c2) c2
# * 54ccb66 (tag: c1) c1
# * baf09b9 (tag: c0, origin/master, origin/HEAD) new Projects Start!!

# Tag 삭제시
$ git tab --delete c4
$ git log --decorate --oneline
# * 073d13c (HEAD, tag: c4", master, iss1) c4
# * a2b1670 (tag: c3) c3
# * e1e72c5 (tag: c2) c2
# * 54ccb66 (tag: c1) c1
# * baf09b9 (tag: c0, origin/master, origin/HEAD) new Projects Start!!
  • 보통 Remote Branch 에 commit 을 깔끔하게 적용하고 싶을 때 사용.

(갈라져 나온 topic branch를 Rebase로 정리해서 master Branch에 merge를 위한 준비)

  • 그냥 참여하는 브랜치에서 메인 프로젝트에 Patch를 보낼 준비가 되면 하는 것.

  • 이렇게 Rebase 하고 나면 프로젝트 관리자는 어떠한 통합작업도 필요없다. 그냥 master 브랜치를 Fast-forward 시키면 된다.

  • Rebase 든 Merge 든 최종 결과물은 같고 commit History만 다르다는 것이 중요.

  • Rebase 브랜치의 변경사항을 순서대로 다른 브랜치에 적용하면서 합치고

Merge의 경우는 두 브랜치의 최종결과만을 가지고 합친다.

Rebase 활용

# --onto : 건너띄어 rebase 하기
# <base-branch> -> <topic-branch-topic-branch>, pass <topic-branch>
git rebase --onto <base-branch> <topic-branch> <topic-branch-topic-branch>


# master -> client, 무시 server
$ git rebase --onto master server client

$ git checkout master
$ git merge client

# git rebase <base-branch-name> <topic-branch-name>
# master -> server
$ git rebase master server

$ git checkout master
$ git merge server

# 사용하지 않는 branch 삭제
$ git branch -d client server

Rebase Risk (위험성)

  • (중요) 이미 Public Repository 에 Push 한 Commit 을 Rebase 하지 마라.

  • Remote Repository 로 보내기 전 Local 에서 Rebase로 commit을 정리해야 함.

Rebase 한 것을 다시 Rebase 하기

  • 다른 누군가 내가 한일을 덮어썼다고 하자. 그러면 내가 했던 일이 무엇이고

덮어쓴 내용이 무엇인지 알아내야 한다.

  • 커밋 SHA 체크섬 외에도 Git은 커밋에 Patch 할 내용으로 SHA 체크섬을 한번 더 구한다.

이 값은 'patch-id' 라고 한다.

  • 덮어쓴 커밋을 받아서 그 커밋을 기준으로 Rebase 할 때 Git은 원래 누가 작성한 코드인지 잘 찾아 낸다. 그래서 Patch가 원래대로 잘 적용된다.
git rebase teamone/master

# config
git config --global pull.rebase true

git pull --rebase <remote-name>

  • 동료가 생성했던 C4와 C4’ 커밋 내용이 완전히 같을 때만 이렇게 동작된다.

  • 커밋 내용이 아예 다르거나 비슷하다면 커밋이 두 개 생긴다(같은 내용이 두 번 커밋될 수 있기 때문에 깔끔하지 않다).

  • 물론 git fetch와 git rebase teamone/master이 두 명령을 직접 순서대로 실행해도 된다.

  • Push 하기 전에 정리하려고 Rebase 하는 것은 괜찮다.

  • 또 절대 공개하지 않고 혼자 Rebase 하는 경우도 괜찮다.

  • 하지만, 이미 공개하여 사람들이 사용하는 커밋을 Rebase 하면 틀림없이 문제가 생긴다.

  • 나중에 후회하지 말고 git pull --rebase.

Rebase vs Merge

  • History 를 변경하지 말아라.

  • 적재적소.

  • Local branch 작업할 때는 History 를 정리하기 위해서 Rebase,

Push로 Remote 밖으로 내보낸 Commit 은 절대 Rebase 하지 말아야 한다.

요약 

# log
git log --oneline --graph --all --decorate  # 이력의 거의 모든 정보 확인

# merge
git checkout <base-branch>
git merge <topic-branch>  # <base-branch>, HEAD <- <topic-branch>
git mergetools  # 충돌시 해결

# branch
git branch <branch-name>       # 생성
git checkout <branch-name>     # 이동 <branch-name> 으로
git checkout -b <branch-name>  # 생성 <branch-name>, 이동 <branch-name> 으로
git branch -vv       # 목록 확인 (-vv 더 자세히)
git branch -d        # 삭제
git branch -D        # 강제 삭제
git branch --merged  # merge 목록만

# remote
git remote show <remote-name>           # 보기
git remote add <new-remote-name> <URL>  # 생성
git remote rename <old> <new>           # 변경: remote-name
git remote set-url <remote-name> <URL>  # 변경: URL
git remote remove <remote-name>         # 삭제

# push
git push <remote-name> <branch-name>  # 개별로도 branch-name 가능
git push <remote-name> <remote-branch-name>:<awesomebranch-name>  #

# fetch
git fetch <remote-name>   # fetch 로 가져온 <pointer-name>
git merge <pointer-name>  # <pointer-name> 형식 == <origin/master>

git fetch <remote-name>   # fetch 로 가져온 <pointer-name>
git checkout -b <new-branch-name> <remote/pointer-name>  # 새로운 branch 생성

# pull (fetch, merge 를 권장)
git pull <remote-name>  # 즉시 remote repositoy 에서 가져오기

#  tracking branch
git checkout --track <remote-tracking-branch>  # 생성

Branch 란?

  • Commit 완료시

    • Commit Objects
    • 메타데이터와 'Root Tree'를 가르키는 Pointer.
    • 현 Staging Area에 있는 데이터의 스냅샷에 대한 포인터, author 나 커밋 메시지 같은 메타데이터, 이전 커밋에 대한 포인터 등을 포함하는 커밋 개체(커밋 Object)를 저장한다.
    • 이전 커밋 포인터가 있어서 현재 커밋이 무엇을 기준으로 바뀌었는지를 알 수 있다.
    • 최초 커밋을 제외한 나머지 커밋은 이전 커밋 포인터가 적어도 하나씩 있고 브랜치를 합친 Merge 커밋 같은 경우에는 이전 커밋 포인터가 여러 개 있다.
    • Tree Objects : Files, Directory structure.
    • 각 Files Blob 객체들
    • Blob : 파일을 Stage 하면 Git 저장소에 파일을 저장.
    • File 마다 생성됨, CheckSum, SHA-1 로 되어있음.
    • 최초 commit 시 master 이름의 Branch 생성됨.

    'master' Branch 는 특별하지 않다.

    git init 명령으로 초기화 때 자동으로 만들어진게 이유가 끝이다. - commit 을 만들 때 마다 Branch가 자동으로 가장 마지막 commit 을 가르킴.

  • 브랜치를 만들어야 하면 프로젝트를 통째로 복사해야 하는 다른 버전 관리 도구와 Git의 차이는 극명하다. 통째로 복사하는 작업은 프로젝트 크기에 따라 다르겠지만 수십 초에서 수십 분까지 걸린다. 그에 비해 Git은 순식간이다. 게다가 커밋을 할 때마다 이전 커밋의 정보를 저장하기 때문에 Merge 할 때 어디서부터(Merge Base) 합쳐야 하는지 안다.

Rebase

  • 한 branch에서 다른 branch로 합치는 방법. (Merge, Rebase)

  • 차이는 Rebase 는 Branch 를 만들지 않음.

  • master branch 로 merge 하기 전 작업 중인 다른 branch 들을 정리 목적으로 사용.

git rebase <base-branch-name>
git checkout <base-branch-name>
git merge <topic-branch-name>

$ git rebase master

(갈라져 나온 topic branch를 Rebase로 정리해서 master Branch에 merge를 위한 준비)

  • 그냥 참여하는 브랜치에서 메인 프로젝트에 Patch를 보낼 준비가 되면 하는 것.

  • 이렇게 Rebase 하고 나면 프로젝트 관리자는 어떠한 통합작업도 필요없다. 그냥 master 브랜치를 Fast-forward 시키면 된다.

  • Rebase 든 Merge 든 최종 결과물은 같고 commit History만 다르다는 것이 중요.

  • Rebase 브랜치의 변경사항을 순서대로 다른 브랜치에 적용하면서 합치고

Merge의 경우는 두 브랜치의 최종결과만을 가지고 합친다.

Rebase Risk (위험성)

  • (중요) 이미 Public Repository 에 Push 한 Commit 을 Rebase 하지 마라.

  • Remote Repository 로 보내기 전 Local 에서 Rebase로 commit을 정리해야 함.

Rebase 한 것을 다시 Rebase 하기

  • 다른 누군가 내가 한일을 덮어썼다고 하자. 그러면 내가 했던 일이 무엇이고

덮어쓴 내용이 무엇인지 알아내야 한다.

  • 커밋 SHA 체크섬 외에도 Git은 커밋에 Patch 할 내용으로 SHA 체크섬을 한번 더 구한다.

이 값은 'patch-id' 라고 한다.

  • 덮어쓴 커밋을 받아서 그 커밋을 기준으로 Rebase 할 때 Git은 원래 누가 작성한 코드인지 잘 찾아 낸다. 그래서 Patch가 원래대로 잘 적용된다.
git rebase teamone/master

# config
git config --global pull.rebase true

git pull --rebase <remote-name>

  • 동료가 생성했던 C4와 C4’ 커밋 내용이 완전히 같을 때만 이렇게 동작된다.

  • 커밋 내용이 아예 다르거나 비슷하다면 커밋이 두 개 생긴다(같은 내용이 두 번 커밋될 수 있기 때문에 깔끔하지 않다).

  • 물론 git fetch와 git rebase teamone/master이 두 명령을 직접 순서대로 실행해도 된다.

  • Push 하기 전에 정리하려고 Rebase 하는 것은 괜찮다.

  • 또 절대 공개하지 않고 혼자 Rebase 하는 경우도 괜찮다.

  • 하지만, 이미 공개하여 사람들이 사용하는 커밋을 Rebase 하면 틀림없이 문제가 생긴다.

  • 나중에 후회하지 말고 git pull --rebase.

Rebase vs Merge

  • History 를 변경하지 말아라.

  • 적재적소.

  • Local branch 작업할 때는 History 를 정리하기 위해서 Rebase,

Push로 Remote 밖으로 내보낸 Commit 은 절대 Rebase 하지 말아야 한다.
