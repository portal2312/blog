Git 저장소 만들기

기존 저장소를 Git 저장소로 만들기

git init    # 현재 경로에 .git 생성
git add <files>
git add LICENSE
git commit -m 'commit msg'

기존 저장소를 Clone 하기

git clone <URL>
git clone <URL> <Dir>    # 원하는 경로, Dir 명으로 저장
git clone <URL>.git <Dir>    # 저장소 추가

수정하고 저장소에 저장하기

vi <file>
git add <file>                    ----------------------------------->
                                  <---------------------------------- git commit
                                  vi <file> --->
                                                 git add <file> ----->
                                  <---------------------------------- git commit
<-------------                    git remove <file>

파일 무시하기

.gitignore

cat .gitignore

*.[oa]    # .o 나 .a 로 시작하는 파일 무시
*~        # ~ 로 시작하는 파일 무시
  • 아무것도 없는 line, 샾(#) 은 무시.
  • 표준 Glob 패턴 사용. <- 'Glob' Unix 계열에서 사용되는 CLI.
  • 슬래시(/)로 시작하면 하위 Dir 에 적용되지(Recursivity) 않는다.
  • Dir는 슬래시(/)를 끝에 사용하는 것으로 표현.

  • 느낌표(!)로 시작하는 패턴의 파일은 무시하지 않는다.

  • Glob 패턴은 정규표현식을 단순하게 만든 것으로 생각하면 되고 보통 쉘에 서 많이 사용한다. 애스터리스크()는 문자가 하나도 없거나 하나 이상을 의 미하고, [abc]는 중괄호 안에 있는 문자 중 하나를 의미한다(그러니까 이 경 우에는 a, b, c). 물음표(?)는 문자 하나를 말하고, [0-9]처럼 중괄호 안의 캐릭 터 사이에 하이픈(-)을 사용하면 그 캐릭터 사이에 있는 문자 하나를 말한다. 애스터리스크 2개를 사용하여 디렉토리 안의 디렉토리 까지 지정할 수 있다. a/*/z 패턴은 a/z, a/b/z, a/b/c/z 디렉토리에 사용할 수 있다.

    • Examples (https://github.com/github/gitignore) ``` # 확장자가 .a인 파일 무시 *.a

# 윗 라인에서 확장자가 .a인 파일은 무시하게 했지만 lib.a는 무시하지 않음 !lib.a

# 현재 디렉토리에 있는 TODO파일은 무시하고 subdir/TODO처럼 하위디렉토리에 있는 파일은 무시하지 않음 /TODO

# build/ 디렉토리에 있는 모든 파일은 무시 build/

# doc/notes.txt 파일은 무시하고 doc/server/arch.txt 파일은 무시하지 않음 doc/*.txt

# doc 디렉토리 아래의 모든 .txt 파일을 무시 doc/*/.txt ```

Staged와 Unstaged 상태의 변경 내용을 보기

git diff             # [Working Dir] - [Staging Area] 비교, not staged
                     # unmodified <> modified
git diff --cached    # git add 이후 상태와 git commit 완료된 내용을 비교
                     # modified <> staged
  • Unmodified 상태의 파일은 비교하지 않는다. (해당 파일이 git commit 이후 즉시)

변경사항 Commit 하기

  • Git은 Staging Area에 속한 Snapshot을 Commit 한다.

  • add 되지 않는 파일은 Commit 하지 않는다.

$ git config --global core.editor vim

  • 둘째 라인부터 git status 명령의 결과가 채워진다.

  • 내용을 저장, 편집기 종료 > new Commit 완성. (#로 시작하는 내용을 제외한)

그냥 종료시 commit 되지 않는다.

  • 수정은 했지만, 아직 Staging Area에 넣지 않은 것은 다음에 커밋할 수 있다. 커밋할 때마다 프로젝트의 스냅샷을 기록하기 때문에 나중에 스냅샷끼리 비교하거 나 예전 스냅샷으로 되돌릴 수 있다.

Staging Area 생략하기

-a 옵션으로 추적중인 파일을 git add 하지 않고 즉시 commit 할 수 있음.

git commit -a -m 'success'

파일 삭제하기

삭제 후 적용하려면 반드시 commit 해야 함.

git rm <file>           # WorkingDir + Staging Area 삭제 (Local + Git)
git rm --cached <file>  # Staging Area 삭제 (Git)
git rm ./logs/\*.log    # Pattern 삭제

파일 이름 변경하기

git mv <file> <new file>


mv <file> <new file>
git rm <file>
git add <new file>

Commit History 조회하기 

git log
  • Options (주요 옵션)
Options Description
-p 각 커밋에 적용된 패치를 보기 (diff 내용 나옴)
--stat 각 커밋에서 수정된 파일의 통계정보 보기
--shortstat --stat 명령의 결과 중에서 파일의 (= 추가, 수정, 삭제) 라인만 보기
--name-only 커밋 정보중에서 수정된 파일의 목록만 보기
--name-status 수정된 파일의 목록과 상태(추가, 수정, 삭제) 보기
--abbrev-commit 40자 짜리 SHA-1 체크섬을 전부 보여주는 것이 아니라 처음 몇 자만 보기
--relative-date 시간을 “2 weeks ago”처럼 상대적인 형식으로 보기
--graph 브랜치와 머지 히스토리 정보까지 아스키 그래프로 보기
--pretty 지정한 형식으로 보기 (옵션: oneline, short, full, fuller, format 등, format은 원하는 형식으로 출력하고자 할 때 사용)
  • --pretty bash git log --pretty=format:"%h - %an, %ar : %s"

| Options | Description | | :------ | :---------- | | %H | 커밋 해시 | %h | 짧은 길이 커밋 해시 | %T | 트리 해시 | %t | 짧은 길이 트리 해시 | %P | 부모 해시 | %p | 짧은 길이 부모 해시 | %an | another 이름 | %ae | another 메일 | %ad | another 시각 (형식은 –date= 옵션 참고) | %ar | another 상대적 시각 | %cn | commiter 이름 | %ce | commiter 메일 | %cd | commiter 시각 | %cr | commiter 상대적 시각 | %s | 요약

  • Example
git log --pretty=format:"%h %s" --graph

git checkout -- <file>

  • git checkout -- 명령은 위험한 명령이

  • 변경한 내용을 쉽게 버릴수는 없고 하지만 당장은 되돌려야만 하는 상황이 라면 Stash와 Branch를 사용하자.

Remote Storage (리모트 저장소)

현재 프로젝트에 등록된 원격 저장소

확인하기

$ git remote -v    # -v: URL 보기

origin  http://localhost/mks/twisted_example.git (fetch)
origin  http://localhost/mks/twisted_example.git (push)

  • origin: clone 시 자동으로 remote name 이 저장됨

  • 참조: 4. Git 서버 > 서버설정하기

추가하기

$ git remote add <remote-name> <URL>

Pull 과 Fetch 하기

$ git fetch <remote-name>

$ git fetch origin   # 마지막으로 가져온 이후 수정된 것을 모두 가져옴, 수동 merge

$ git pull origin    # 마지막으로 가져온 이후 수정된 것을 모두 가져옴, 자동 merge

  • git fetch : 리모트 저장소의 데이터를 모두 로컬로 가져오지만, 수동 Merge 해야함.

  • git clone : 자동으로 리모트 저장소를 'origin' 이름으로 추가됨.

Push 하기

$ git push <remote-name> <branch-name>

$ git push origin master

저장소 살펴보기

$ git remote show <remote-name>

* remote origin
  Fetch URL: http://localhost/mks/twisted_example.git
  Push  URL: http://localhost/mks/twisted_example.git
  HEAD branch: master
  Remote branch:
    master tracked
  Local branch configured for 'git pull':
    master merges with remote master
  Local ref configured for 'git push':
    master pushes to master (fast-forwardable)

저장소 명/URL 변경하기

git remote rename <old-name> <new-name>

git remote set-url <remote-name> <URL>

Tag

  • Tag 삭제하기 bash git tag --delete <tag-name>

조회하기

git tag

git show <tag-name>

추가하기

  • Lightweight : 단순 commit pointer.

  • Annotated : 아래 내용을 저장할 때만 추천.

    • Git DB에 Tag의 create Name, Email, create Date, tag Msg.
    • GPG(GNU Privacy Guard)로 서명.

Annotated Tag

git tag -a <tag-name> -m 'description'

git tag -a v1.0 -m 'first version'

-a : annotated tag

Lightweight Tags

git tag v1.1-lw    # <tag-name-lw>
  • -a, -s, -m 옵션을 사용하지 않음.

나중에 Tag 하기

git tag -a <tag-name> checksum
  • Example
  $ git log --pretty=oneline

  806e37cb64cda5a59f721e56190929a22dbd3af1 add file3.txt
  226ae8bbcabea3fc9cc3ad2ca7e710b265ff7e96 ADD file3.txt
  abd6c86cd73f19554fd64b982cd8396e91206906 delete file4.txt
  76ec8b5142f5e95e4ac7f7588277b17cf3f61a3a add file4.txt

  git tag -a v1.0 806e37cb64cda5a59f721e56190929a22dbd3af1 -m 'first version'

Tag 공유하기

git push <remote-name> <tag-name>    # 한개씩

git push <remote-name> --tags        # 여러개 한번에

$ git push origin v1.0
  • 수동, 별도로 tag 를 전송해야 함. (git push 동시에 안됨)

Tag를 Checkout 하기

  • tag는 branch 와는 달리 가리키는 commit을 바꿀 수 없는 이름이기 때문에 Checkout 해서 사용할 수 없다.

  • tag가 가리키는 특정 commit 기반의 branch를 만들어 작업하려면 새로 브랜치를 생성해야 함.

Git Alias 

git config --global alias.chk checkout

git config --global alias.last 'log -1 HEAD'
  • vi ~/.gitconfig 목록을 볼 수 있다.

파일 무시하기

.gitignore

cat .gitignore

*.[oa]    # .o 나 .a 로 시작하는 파일 무시
*~        # ~ 로 시작하는 파일 무시
  • 아무것도 없는 line, 샾(#) 은 무시.
  • 표준 Glob 패턴 사용. <- 'Glob' Unix 계열에서 사용되는 CLI.
  • 슬래시(/)로 시작하면 하위 Dir 에 적용되지(Recursivity) 않는다.
  • Dir는 슬래시(/)를 끝에 사용하는 것으로 표현.

  • 느낌표(!)로 시작하는 패턴의 파일은 무시하지 않는다.

  • Glob 패턴은 정규표현식을 단순하게 만든 것으로 생각하면 되고 보통 쉘에 서 많이 사용한다. 애스터리스크()는 문자가 하나도 없거나 하나 이상을 의 미하고, [abc]는 중괄호 안에 있는 문자 중 하나를 의미한다(그러니까 이 경 우에는 a, b, c). 물음표(?)는 문자 하나를 말하고, [0-9]처럼 중괄호 안의 캐릭 터 사이에 하이픈(-)을 사용하면 그 캐릭터 사이에 있는 문자 하나를 말한다. 애스터리스크 2개를 사용하여 디렉토리 안의 디렉토리 까지 지정할 수 있다. a/*/z 패턴은 a/z, a/b/z, a/b/c/z 디렉토리에 사용할 수 있다.

    • Examples (https://github.com/github/gitignore) ``` # 확장자가 .a인 파일 무시 *.a

# 윗 라인에서 확장자가 .a인 파일은 무시하게 했지만 lib.a는 무시하지 않음 !lib.a

# 현재 디렉토리에 있는 TODO파일은 무시하고 subdir/TODO처럼 하위디렉토리에 있는 파일은 무시하지 않음 /TODO

# build/ 디렉토리에 있는 모든 파일은 무시 build/

# doc/notes.txt 파일은 무시하고 doc/server/arch.txt 파일은 무시하지 않음 doc/*.txt

# doc 디렉토리 아래의 모든 .txt 파일을 무시 doc/*/.txt ```

Staged와 Unstaged 상태의 변경 내용을 보기

git diff             # [Working Dir] - [Staging Area] 비교, not staged
                     # unmodified <> modified
git diff --cached    # git add 이후 상태와 git commit 완료된 내용을 비교
                     # modified <> staged
  • Unmodified 상태의 파일은 비교하지 않는다. (해당 파일이 git commit 이후 즉시)

Remote Storage (리모트 저장소)

현재 프로젝트에 등록된 원격 저장소

확인하기

$ git remote -v    # -v: URL 보기

origin  http://localhost/mks/twisted_example.git (fetch)
origin  http://localhost/mks/twisted_example.git (push)

  • origin: clone 시 자동으로 remote name 이 저장됨

  • 참조: 4. Git 서버 > 서버설정하기

추가하기

$ git remote add <remote-name> <URL>

Pull 과 Fetch 하기

$ git fetch <remote-name>

$ git fetch origin   # 마지막으로 가져온 이후 수정된 것을 모두 가져옴, 수동 merge

$ git pull origin    # 마지막으로 가져온 이후 수정된 것을 모두 가져옴, 자동 merge

  • git fetch : 리모트 저장소의 데이터를 모두 로컬로 가져오지만, 수동 Merge 해야함.

  • git clone : 자동으로 리모트 저장소를 'origin' 이름으로 추가됨.

조회하기

git tag

추가하기

  • Lightweight : 단순 commit pointer.

  • Annotated : 아래 내용을 저장할 때만 추천.

Annotated Tag

git tag -a <tag-name> -m 'description'

git tag -a v1.0 -m 'first version'

-a : annotated tag

Lightweight Tags

git tag v1.1-lw    # <tag-name-lw>
  • -a, -s, -m 옵션을 사용하지 않음.
